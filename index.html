Linux Security

bpf

场景 1：用户态的 tcpdump，需要过滤网络包，但是每次内核只做转发，需要每个网络包 copy 到用户态 效率太低，最好内核根据某个条件自由过滤网络包。

场景 2：seccomp 限制系统调用，根据定死的规则限制太不灵活，最好有一个复杂的逻辑根据参数判断是否需要过滤，由用户态提供。

本质：

  1. 操作系统内核向下提供对硬件的调用，向上对用户态提供 api，同时涉及到内存管理、调度等重要功能，但是在具体的场景和需求中，仍然是需要由用户态来决定业务逻辑的。
  2. 内核权限更高、接触数据更底层，在这里运行用户态的逻辑更有价值。
  3. 内核本身不应该随意修改，且加入大量复杂的代码。

classic-bpf/ extent bpf：

  1. 内核提供一些入口，可以运行 bpf 程序
  2. 用户态写 bpf 程序，挂载到内核运行
  3. 内核检查，然后在挂载点运行

具体实现：

  1. bpf 程序可以用一些限定的数据结构，一些寄存器，内存。相当于是另一套小的指令集（RISC ISA）这套指令集会通过解释器或者编译器翻译成具体的汇编执行。
  2. 解释执行->编译 JIT，修改挂载点的第一个指令为 int3 ->内核编译期使用 fentry
  • bpf 的挂载点提供 context 可以访问内存数据
  • 提供一些 helper 函数

GPU Travelling: Efficient Confidential Collaborative Training with TEE-Enabled GPUs

GPU Travelling: Efficient Confidential Collaborative Training with TEE-Enabled GPUs

CCS’25

Background

多个彼此不信任的数据持有方，希望一起训练一个模型，但又不能泄露各自的私有数据，原因可能是隐私、法规、版权或商业竞争。

随着现代训练任务尤其是 LLM 的模型和数据集越来越大，现有方案在训练过程中需要频繁交换数据或模型，而这些交换通常走的是较慢的常规链路，通信成本会变得非常高。

Transferring model

中心节点先把基础模型发给各数据方，各自用本地数据训练，再把梯度或模型参数传回中心聚合。

问题：

  • 模型越大，梯度/参数更新越大，通信开销很高
  • 每个数据方都必须拥有足够的本地算力和显存来训练整模型

Transferring dataset

把各方数据集传到一个中心训练节点，在那里做统一训练，并依靠 TEE 等机制保护数据机密性。

然而数据持有方是不希望自己的数据是被集中保存的，本文考虑的场景是，数据训练方和数据存储方不在同一个物理节点上

每次训练需要用到不同的数据集时，需要通过网络进行传输，效率低

Confidential Computing

传统的 CVM 方案是不信任设备的，但是 NVIDIA 提出了 GPU TEE 可以与 CVM 例如 Intel TDX 协同工作

GPU 和 CVM 的驱动会协商一个密钥用于通信，CVM 将 private memory 加密后放置于 bounce buffer，GPU DMA 后用对应的 key 进行解密，然后放入 VRAM

Insight

Never underestimate the bandwidth of a station wagon full of tapes hurtling down the highway
——Andrew S. Tanenbaum

永远不要低估一辆载满磁带的在高速公路上飞驰的卡车的带宽

多个数据提供方把数据提前传到处于一台连接了 GPU 的物理机的多个 CVM 中

GPU 所有权在多个 CVM 中切换，数据通过 PCIe 拷贝到 GPU 上，而不是通过网络

orchestrator 是负责训练的 可信的 CVM

Challenge

  1. 现有的 GPU 架构设计中，如果让 GPU 在不同 CVM 间切换，会去除所有的数据（模型也会被丢掉）
  2. CVM 与 GPU 的通信依赖协商好的 key、Initial Vector 以及正确的 driver 上下文状态。

Design

CVM Roles

  • One Orchestrator：可信的中心节点，在这个节点上进行训练。训练过程中需要发起请求，让数据提供者拷贝数据到 GPU 上
  • Multiple Data Holder：私有数据提供者，通过远程验证信任 Orchestrator 与 GPU 的 CC 环境，随后在训练过程中提供拷贝数据的服务。

PCIe MUX

负责在硬件上把一个 GPU 动态切换给不同的 CVM 使用

  • 中断：将中断路由到不同的 CVM 中
  • DMA mapping：配置 IOMMU 页表

Confidential Context

GPU 会维护自己的 key 和 IV 等状态，需要和 CVM 中 driver 的状态一致，才可以正常通信。

在 CVM 的 GPU driver 中增加 export 和 import 接口，可以导入/导出 key IV 以及 driver context 等信息。

当 GPU 在 CVM 之间切换的时候，上一个 CVM 需要 export，下一个 CVM import，使得下一个 CVM 可以正常与 GPU 通信。

为了上一个 CVM 无法使用 key 窃取下一个 CVM 的数据，在进行了一次切换后，会进行 key rotation 更新 key。

Evaluation

  • dual Intel Xeon Silver 4516Y+ 2.2GHz 24-Core CPUs , 256 GiB of RAM
  • a NVIDIA H100 GPU
  • Both the CVMs and the hypervisor run Ubuntu 24.0.1 LTS.
  • CVMs are protected under Intel TDX with a stock Linux Kernel 6.8.12 4 CPU cores, 8 GiB of RAM

Breakdown

主要开销如下：

  • PCIe MUX 切换，单次 150ms
  • Read Dataset，baseline 也需要，不算 GPU travelling 的 overhead
  • GPU write，baseline 也需要，不算 GPU travelling 的 overhead
  • SSL overhead：export/import key 中 CVM 和 GPU 通信 的耗时，稳定在 40ms

Benchmarks

测试不同网络带宽、不同 buffer size 下的数据传输时间

在 512M 带宽 4M 的情况下，GPU travelling 取得高达一千倍的优势

只要 buffer size>4M 或者 bandwidth < 500M GPU travalling 都会优于 baseline

在实际应用中

  • GPU VRAM 80G 不会每次只分配 4M
  • 不会给每个 data holder 高达 500M 的带宽

Application

llm.c: an open-source CUDA-based LLM training project using the GPT-2 model

测试 fine-tuning 过程的耗时

GPU travelling 压缩了训练过程中的数据传输时间，占用训练时间的占比不超过千分之一，且与网络带宽无关

Outlooks

本文场景目前的限制是，所有 data holder 所在的 CVM 必须在同一个连接 GPU 的物理机器上。

Hardware PCIe switches (PCIe fabric)

使用 PCIe switch 在物理上支持将一个 GPU 连接到不同的物理机上使用

broadcom 已有具有此功能的产品

可扩展至数据中心级别的场景，而非单个物理机

Conclusion

本文提出了 GPU Travelling，通过允许 GPU 安全地切换至数据持有者处以收集数据集，消除了通过较慢连接传输加密大型数据集或模型的需求，利用支持 TEE 的 GPU 实现高效的机密协同训练。对于常见的数据缓冲区大小和网络带宽，GPU Travelling 优于传统方法，并且能够为像 LLM 这样的现代协同训练工作负载在相对性能和绝对时间节省方面带来显著收益。

RKNPU driver

linux内核驱动：https://github.com/airockchip/rknn-llm/tree/main/rknpu-driver

User

用户态使用方式：https://github.com/mtx512/rk3588-npu

ioctl

用户态通过内核接口申请内存

作用是分配一段dma buffer

  • 这块内存可以被CPU访问，填入需要计算的数据
  • 这块内存地址可以被提交给NPU，NPU通过dma获取数据 并且写入计算后的结果

data

提交给NPU的数据需要重排

C是总列数 必须是32的倍数

内存连续排列为原有数组的：

0 0/0 1/0 2/…/0 31/1 0/1 1/1 2/…/1 31/2 0/2 1/2 2/…/2 31/ … / 31 0/31 1/31 2/…/31 31/ 一共32x32=1024个一组

然后是0 32/0 33/0 34/…/0 63/1 32/1 33/…/1 63./…/ 31 63/ 1024个

一共列/32组 每组1024个 一共32*1024个

所以pkg32C是跳过这些

这些后面紧跟着 是32 0/ 32 1/ 32 2/ …/ 32 31/ 33 0/ 33 1/ 33 2/../ 33 31

列数不会超过C cpg一定小于C/32 因此cpg*32严格小于C 不会重复

总的顺序是 先按32份块

  • 每小块 连续一行32个 然后第二行 一共32行 1024个
  • 不同块 按列递增 第一块的1024个之后 跟着第一块右边的 也就是列+32的第二块 直到列到头
  • 列到头以后 行+32 列归零 还是按列递增 列到头再按行

这里驱动分配内存需要完成的工作如下：

  1. kzalloc分配一个内核对象，这个rknpu task的内核对象只是内核自己用，会记录一些元数据。
  2. dma_alloc_attrs分配dma buffer，对于dma buffer的要求是size大小的连续内存，这个函数的返回值为kvaddr 也就是内核使用的虚拟地址，真正的dma addr记录在buf->dma_addr。对于dma addr在开启iommu的时候是iova，不开的时候就是pa。
  3. mmap，把这块dma buffer对应的内存映射给用户态，因为用户态不能直接用内核态的虚拟地址。

这里分配的dma buffer默认应该会通过页表的某个bit控制缓存行为，是写穿到内存的，否则会出现CPU把数据写在CPU缓存内，NPU访问主存读到旧数据的情况。

submit

通过MMIO写入rknpu硬件寄存器

  • RKNPU_OFFSET_INT_MASK用于开启中断，rknpu任务结束后硬件会给CPU发送中断
  • RKNPU_OFFSET_PC_DATA_ADDR用于写入最重要的任务地址，regcmd_addr是regcmd这个dma buffer的地址，这个dma buffer记录了这次npu计算任务的元数据，包括input weight output的dma addr。

虽然原版代码没有提供，但是实测后发现可以不依赖中断，而是通过这种方式轮询获取任务完成的信息。

probe

arch/arm64/boot/dts/rockchip/rk3588s.dtsi

这里可以看到 rknpu和rknpu_mmu的中断号是一样的，并且rknpu_mmu的mmio区域是rknpu的子集

对于rockchip rknpu_mmu是只给rknpu用的，其他设备有其他设备自己的iommu可以用

probe太长了 主要关注

  1. 注册MMIO base，把物理地址0xfdab0000映射到内核虚拟地址。
  2. 注册中断，一共3个核，每个核一个中断，中断处理函数为rknpu_irq_handler
  3. 初始化rknpu_iommu
  4. rknpu_power_on与rknpu_power_off 给设备上电

这里设备的管理是使用完一次就延迟3s后断电

实测发现在没有上电的情况下进行MMIO，设备会直接panic重启。

IOMMU

iommu的初始化在drivers/iommu/rockchip-iommu.c

对于rockchip的iommu都是使用同一套代码，所以这个文件中的probe等函数会被执行多遍，但每一遍其实是针对不同的设备。

其中存在rk_iommu_irq，也就是说当发生中断时，除了rknpu_irq_handler会被调用，rk_iommu_irq也会被调用。

实测rk_iommu_irq没有什么用，去掉也能照常运行。

rknpu的iommu在dtsi中配置为打开，所以会默认会通过iommu进行一次地址翻译。

因此在分配完dma buffer以后，需要分配一个IOVA，把IOVA-PA的映射写到iommu的页表中。

http://pro934ad4.pic3.ysjianzhan.cn/upload/Rockchip_Developer_Guide_Linux_IOMMU_CN.pdf

虽然rk3588支持40bit dma索引，但是iommu只有2级页表，支持32bit的dma addr，故打开iommu的情况下，rknpu仍然只支持4G地址索引。

drivers/rknpu/rknpu_iommu.c提供了一些封装好的方法 主要做几件事

  1. 用内核接口例如sg_xx dma_xx 分配内存
  2. 自由进行iova的分配
  3. 用iommu接口进行iommu页表配置、domain切换等

interrupt / job schedule

rknpu_job.c中实现了调度的逻辑

每次用户进行submit的ioctl时，不直接进行MMIO，而是写到内核自己的job队列中

发生中断后，表示某个NPU核上的任务完成，可以调度下一个任务

用户提交rknpu job时，可通过RKNPU_JOB_NONBLOCK控制是否阻塞等待

内核会根据情况调用rknpu_job_wait 进入循环调用wait_event_timeout 等待RKNPU_JOB_DONE 否则让出cpu

Passthrough

把RKNPU直通进虚拟机

  1. host提前分配dma buffer，建立iommu映射，然后把dma buffer对应的HPA映射给GPA，并且分配IOVA=GPA
  2. host把mmio base 0xfdab0000映射给host
  3. host unregister rknpu与rknpu iommu的中断，可以重新register一个然后通过eventfd 在kvm中注入中断，但是实际上如果不需要job调度 可以通过轮询的方式，是不需要中断的。
  4. guest中可通过open(/dev/mem,O_SYNC) mmap的方式拿到GPA对应的dma buffer，然后填入IOVA=GPA
  5. MMIO提交给npu，此时npu会照常完成运算并写入结果。

Conclusion

NPU driver需要完成

  1. 通过内核接口进行dma内存分配，并且给用户态使用。
  2. 配置中断，实现任务提交与调度逻辑。
  3. 在有IOMMU等情况下，与IOMMU配合，分配IOVA建立页表。
  4. 对设备的电源、状态检查与管理。

NVMePass: A Lightweight, High-performance and Scalable NVMe Virtualization Architecture with I/O Queues Passthrough

Posted on 2026-02-20 Edited on 2026-02-22 In papers Word count in article: 1.1k Reading time ≈ 4 mins.

NVMePass: A Lightweight, High-performance and Scalable NVMe Virtualization Architecture with I/O Queues Passthrough

HPCA’25

背景

设备虚拟化

  1. 软件虚拟化：
    • 全虚拟化 纯qemu模拟 性能差
    • 半虚拟化 virtio 性能差 vmexit 上下文切换 50%overhead
    • 基于轮询的虚拟化 spdk-vhost 轮询硬件队列 需要额外CPU资源，当虚拟机数量太多的时候延迟高，浪费12%cpu资源
  2. 硬件虚拟化：
    • VFIO 依赖硬件MMIO 不可共享 只能给一个虚拟机
    • SR-IOV 依赖硬件PF-VF 宿主机管理PF 每个VF给虚拟机 对硬件要求高 成本高

NVMe

https://www.dingmos.com/index.php/archives/152/

The NVMe specification allows one Admin Queue Pair to manage the entire controller, handling
functions like creating I/O Queue Pairs and namespace management. Up to 65,535 I/O Queue Pairs can be associated with each Admin Queue Pair, specifically for executing I/O commands(read/write). Each Queue Pair (QP) consists of a Submission Queue (SQ) for submitting commands and a Completion Queue (CQ) for receiving completions. SQ and CQ are circular buffers in host memory, shared with the device through DMA. The head or tail pointer of each queue is maintained in its Doorbell Registers (DBs). Furthermore, Controller Memory Buffer is a general-purpose read/write memory region on the NVMe device controller that can be utilized for various purposes.

读写过程：

  1. kernel确定读写的内存地址 即DMA地址 以及数据在设备上的地址 即LBA
  2. 将信息写入SQ/CQ
  3. 写doorbell 寄存器 （MMIO）提交请求

设计

目标：

  1. 性能好 接近原声
  2. 不浪费CPU资源
  3. 可扩展性好，虚拟机密度大时不影响性能
  4. 隔离与安全性好，多个虚拟机的数据不会互相泄漏

架构和传统的virtio设备一样：

  1. 虚拟机内前端 不需要修改用户态程序
  2. host/hypervisor上的后端和设备驱动
  3. 固件中的保护和检查代码

控制平面：vmexit到host处理 不在关键路径上

数据平面：通过EPT映射IO队列 直接直通。每个虚拟机的LBA和HPA配置的时候都分开

  • 直通：虚拟机驱动中的IO队列先不映射到HPA，自然触发EPT page fault，然后映射到host上IO队列的HPA 完成直通
  • 隔离：每个VM拥有完全属于自己的IO队列、DMA范围、寄存器MMIO地址、LBA
    • IO队列：host的IO队列有很多（65535个），可以直接分给不同的虚拟机，EPT隔离。这是整个设计中最关键最巧合的地方，因为NVMe硬件碰巧支持这么多。
    • DMA地址：每个VM使用不同的HPA范围，EPT隔离。然而NVM硬件的IOMMU只有一个，这篇文章的操作是每个VM使用完全不同的GPA范围，这样IOMMU自然可以翻译到不同的HPA范围。下文详细解释。
    • doorbell寄存器（MMIO）：配置CAP.DSTRD可控制偏移，不同VM使用不同物理页 EPT隔离
    • LBA：先hypervisor划分范围，但vm仍然可以伪造其他vm的LBA提交给硬件，所以在NVM固件中加入检查

相当于把硬件提供的所有能力全部在软件上手动划分开，每个固定范围给一个虚拟机用，然后通过EPT、IOMMU、NVM固件来检查和隔离。

PRP/SGL其实就是写的DMA address，也就是说 在打开IOMMU的情况下，NVMe硬件会自动通过IOMMU把PRP中的信息认为是IOVA（GPA）翻译为HPA，此时是不需要固件去保护隔离的。

设备只有一个，IOMMU同时只能用一张页表。如果每个虚拟机用自己的页表，IO的时候切换IOMMU的页表，那么就无法完成并行读写，可扩展性差，所以引发了本文的设计：共享IOMMU页表。所有VM的GPA段完全不重复，使得IOMMU的一张页表可以完成所有虚拟机的翻译。

存在两个问题。

  1. 是否影响VM内核本身的逻辑？还是只改一个PA_BASE就可以？
  2. 某种程序上约束了可扩展性，因为一个VM需要的DMA地址范围*虚拟机数量一定要小于宿主机内存大小。

SCAVY: Automated Discovery of Memory Corruption Targets in Linux Kernel for Privilege Escalation

Usenix Security‘24

https://github.com/BadDataLab/SCAVY

https://xz.aliyun.com/news/19324

背景

  1. CVE提出的内存漏洞，例如UAF等，都太小，不足以完成一次提升权限的利用
  2. 一些利用方法 受到传统保护机制 例如SMEP的影响 成功率不高
  3. 因为没有端到端的严重威胁 所以某些CVE并不会受到重视 立即被开发者修复

本文找内核关键数据结构 可以被CVE利用达到权限提升的目的

linux漏洞利用思路

  1. fuzz 找一些内存越界、UAF等导致的crash场景
  2. 找一些这个内存corrupt进一步可拥有的capability
  3. 组合多个POC 完成权限提升
  4. 提高稳定性（触发概率）与能力（绕过更多的安全措施）

本文目标找到更多的memory target可以给2 3用，不参与1 4

一个例子：vm_area_struct

  1. mmap映射/etc/passwd和另一个普通文件，内核分配两个vm_area_struct
  2. 通过一个已经存在的CVE泄漏/etc/passwd对应的vm_area_struct的地址，覆盖普通文件的vm_area_struct的值
  3. msync写回，此时文件是/etc/passwd 但权限变为可写
  4. su提权

千问对于本文工作的解释：

🎯 核心目标

自动找出内核中哪些内存区域被破坏后，能让普通用户获得 root 权限。

这比传统模糊测试（fuzzing）更进一步——不仅要找崩溃（crash），还要判断这个崩溃是否“有用”（即能否提权）。

一、为什么传统方法不够？

  • 像 Syzkaller 这样的模糊测试器擅长触发各种代码路径（高覆盖率），但不关心“破坏哪个字段才有用”。
  • 像 KOOBE / FUZE 能检测内存越界或破坏，但无法判断破坏后是否能提权。
  • 手动分析崩溃是否可利用非常耗时。

👉 SCAVY 的创新点：把整个利用过程拆成三个阶段，每个阶段用不同的策略精准打击。

二、漏洞利用的“三阶段模型”

SCAVY 观察到真实提权漏洞通常分三步：

阶段 目标 例子
1. 分配目标对象 找到内核中可被控制分配的结构体 创建文件 → 分配 struct file
2. 破坏关键字段 修改该结构体中的某个字段（如权限、指针） 把 file->f_cred->uid 改成 0（root）
3. 触发提权行为 使用被破坏的对象执行特权操作 用被篡改的文件句柄写 /etc/passwd

SCAVY 就围绕这三个阶段设计了对应的自动化技术。

三、关键技术详解

🔧 阶段 1：找到“可分配的内核结构体”（Allocator Discovery）

问题：内核有成千上万个结构体，哪些能被用户通过系统调用分配？

SCAVY 的做法：

  • 在内核中插桩（instrument）所有内存分配函数（如 kmalloc）；
  • 同时记录类型转换（比如 (struct task_struct *)kmalloc(...)），从而知道“这块内存是什么类型”；
  • 用 Kprobe 把分配地址 + 类型信息传到用户空间；
  • 模糊测试器遍历系统调用，收集所有“用户能触发分配的结构体”。

✅ 优势：不用污点分析，轻量高效，可扩展到百万级测试用例。

🔍 阶段 2：找出“可被破坏的关键字段”（Field Discovery）

问题：一个结构体有几十个字段，破坏哪个才有用？

SCAVY 的做法：

  • 对每个字段（4~8 字节）注入随机值；
  • 用 硬件观察点（watchpoint） 监控：这个字段是否被读取？
  • 如果被读了 → 说明它“可能影响程序逻辑” → 标记为潜在目标。

💡 为什么有效？
因为如果字段从未被使用，破坏它毫无意义；一旦被读，就可能改变控制流或权限判断。

⚔️ 阶段 3：验证“破坏后能否提权”（Privilege Escalation Detection）

这是最核心的创新！

挑战：

  • 很多破坏会导致立即崩溃（“过早崩溃”），来不及执行提权操作；
  • 如何判断“这次崩溃真的能提权”？

SCAVY 的解决方案：三次对比执行（Differential Analysis）

对同一个测试用例，运行三次：

运行 内存状态 目的
Run A 正常（无破坏） 基线：所有特权操作应失败
Run B 注入随机值破坏 测试是否产生异常行为
Run C 注入从 root 进程复制的有效值 模拟“完美伪造”，看能否成功提权

然后比较三者的：

  • 系统调用返回值（如 write() 是否成功）
  • 输出数据（如读到的文件内容）
  • 访问被破坏内存的指令
  • 代码覆盖率（用 Jaccard 相似度 >0.9 判同）

✅ 如果 Run C 和 Run A 行为不同（比如原本失败的 write 成功了）→ 判定为可提权！

举个例子：

  • 正常时写 /etc/passwd 失败（权限拒绝）；
  • 破坏 file->f_cred 后，写成功了 → 说明获得了 root 权限！

核心就是替换某个kernel object的某个field为root所在进程的对应值，看权限是否发生变化

传统方案只能检测crash，或者监测到能利用的比较少，本文的方法能检测到的可利用位置更多

本文自动化检测出的位置，仍然需要人工验证

  1. 某个object的某个field 的确能导致权限提升。（实际上存在false positive）
  2. 进一步，利用真实的CVE，完成对于某个object的某个field的覆盖和利用，真正达到提权的效果

SLUBStick: Arbitrary Memory Writes through Practical Software Cross-Cache Attacks within the Linux Kernel

Usenix Security‘24

https://github.com/isec-tugraz/SLUBStick/tree/main

背景

  1. 传统的内存漏洞，比如UAF DF OOB 都需要依赖更进一步的方式完成利用（提权）。需要依赖Linux slab、buddy system分配内存的模式，利用成功率不高，然后很容易被修复（对高版本linux无效）
  2. slab分配器，会在空闲对象过多的时候 把这块完整的内存页还给buddy system

设计

整体利用思路：

  1. 用传统内存漏洞构造一个悬挂指针的情况（所在内存页已经归还 但是仍然可以写数据）
  2. 释放所在的slab 直到还给buddysystem
  3. 让这个可被写的对象所在内存页重新分配为用户态页表页
  4. 修改用户态页表页 完成任意地址映射

挑战：

  1. 如何稳定回收并复用这个page
  2. 怎么把准备好的数据通过悬垂指针写到这个page上
  3. 怎么改映射达成任意地址读写

利用过程详细解释

  1. free某个object 如何做到这个object所在的page会还给buddy system？
  • 拥有这个object所在的slab上的其他所有object，同时释放其他所有object
  1. 如何拥有一个victim object所在slab上的其他所有object？
  • 通过open(‘/dev/xxx’)反复申请并占用 阅读源码可以找到不同size的都有
  1. 如何判断这些从内核拿到的object会和victim object在一个slab上？
  • 通过add_key这个系统调用测量时间，当一个slab消耗完以后，从buddy system再拿新的slab会明显变慢。add_key故意失败 会一次malloc+一次free 可重入，没有其他时间干扰 噪声小。因此一次add_key+一次open就可以得知open拿到的内核object是否和上一次在同一个slab上
  1. victim object所在page还给buddy system后 如何再次分配这个页面？
  • buddy system是LIFO的，再次申请这个大小即可。使用mmap+memset 这个page会被分配为PUD页表页
  1. 如何修改页表？
  • 利用悬垂指针。用有漏洞的内核模块和正常的内核模块构造出正常内核object所在内存页被free但是仍然可以通过ioctl访问服务的情况，在服务中copy一份伪造的PUD页表页数据。
  1. 利用悬垂指针修改页表需要利用内核本身提供的服务，限制多，需要保证两个时间窗口的满足。1 kmalloc到copy_from_user之间，需要free所有slab到buddy system完成 否则PUD页不一定是悬垂指针指向的页。 2 完成服务到free obejct时间需要足够长 否则没有完成后续的攻击 这个PUD页就被回收 修改数据了。
  • 使用FUSE，自己写fuse 在fuse_read中加入sleep，然后挂载到某个文件夹，copy_from_user传入文件fd读数据，在发生读的时候会到用户态FUSE控制。
  1. 伪造的PUD可能只有8byte，也就是一个PTE有效，只能访问1G物理内存，且不可重入。
  • 再次大范围mmap+memset，然后检查这1G物理内存的内存数据，如果发现之前不是PTE但之后是PTE，说明这个page变成了最后一级页表页，修改这个PTE即可完成对容易物理地址的映射。
  1. 修改页表，如何确定/etc/passwd所在物理内存？
  • /etc/passwd权限是可读的，读了以后存在在内核的page cache。page cache都是完整的页，直接比较前若干的字符即可
  • 遍历pfn 16G内存可以全部遍历比较，循环总体时间在可接受范围内(16G/4K=4M 4M*1us=4s) 总体可控制在分钟量级
  • 最后达到的效果其实是，可改页表映射整个地址范围，然后任意物理地址读

BUDAlloc: Defeating Use-After-Free Bugs by Decoupling Virtual Address Management from Kernel

Posted on 2026-02-12 In papers Word count in article: 1.5k Reading time ≈ 5 mins.

BUDAlloc: Defeating Use-After-Free Bugs by Decoupling Virtual Address Management from Kernel

Usenix Security 24

https://github.com/casys-kaist/BUDAlloc/tree/main

背景

UAF：

p1 p2是不同模块 使用的内存是同一块，所以仍然可以通过p1访问p2 修改内核关键数据结构提权

解决方案：OTA

In a basic OTA implementation, each object is allocated on a separate page, which is unmapped when the object is freed

p1 free以后 在操作系统上没有限制 理论上应该立即unmap 这样访问p1就会被page fault拦截

但是malloc粒度小，unmap必须4K粒度

解决方案：virtual alias

VA PA中 分配的object仍然是连续的

但是会在VA中为每个object开辟单独的内存页 返回alias地址给用户用 实际上映射的还是PA

但是这个每个object单独的页就可以被方便地unmap

举例

VA中 p1 p2 p3的object分别为0xf000 0xf008 0xf010

PA中 分别为0x1000 0x1008 0x1010

创建三个别名

VA1 = 0xf1 0000

VA2 = 0xf2 0000

VA3 = 0xf3 0000

建立映射

VA 0xf000 - PA 0x1000

VA1 0xf1 0000 - PA 0x1000

VA2 0xf2 0000 - PA 0x1000

VA3 0xf3 0000 - PA 0x1000

p1 = VA1

p2 = VA2 + 0x8

p3 = VA3 + 0x10

达到很好的效果：

  • 物理地址是不影响 不浪费的 不会有碎片
  • 虚拟地址空间需要很大 但是本身是虚拟地址所以无所谓
  • 每个小object被free后都可以立即unmap

存在的问题：

  1. malloc的时候需要syscall一次 不然没法传递给内核map的信息
  2. free的时候需要syscall一次 立即unmap防止UAF
  3. map/unmap涉及到内核的锁 串行化

导致性能差

BUDAlloc设计

  1. 用户态用一个trie树记录alias的映射，不立即mremap，而是等一次page fault。page fault时 内核从共享的trie树中找地址。实际上可以prefault 一次page fault handler中根据metadata把之后的都补上映射
  2. 延迟unmap。每次free后只是记录metadata，等到下一次page fault再batch处理。 一定无法保证正确性，但实践中发现，这个window非常小，已经可以保护几乎所有CVE的UAF场景。
  3. 用eBPF提供page fault handler
  4. 用户态决定一个伪PA，kernel再把伪PA翻译成真PA，这里把并发的锁控制从内核的buddy system等转移到了用户态，用户态可以有更多的语义信息和并发控制算法，提高可扩展性

天然具有的兼容性：

  1. On demand paging
  2. Copy on write
  3. proc file system

兼容性存在问题的地方：

  1. reverse map
  2. PTE ref count

（工程型问题，都解决了）

对应用程序透明，可执行二进制文件是不需要修改的

Evaluation

关注如下几点：

  1. UAF CVE是否完全防御
  2. 性能
  3. 多核性能 可扩展性
  4. memory overhead
  5. 程序兼容性

PipeLLM: Fast and Confidential Large Language Model Services with Speculative Pipelined Encryption

背景：

  1. 英伟达 GPU 也提供了机密计算 类似CPU侧的TDX SEV 提供一个host不可访问的enclave 避免被攻击偷数据
  2. CVM中对private memory 也就是被CPU加密的私有内存调用cuda memcpy 会自动先出发CPU侧的AES加密 然后放到CVM shmem中 给GPU DMA 然后GPU解密。 加密用到的IV是两个硬件初始化的时候自动生成的 不需要每次通信。
  3. 在大模型推理 训练服务中，CPU GPU数据传输过程中的加解密存在关键路径上，overhead很大（由于GPU内存比模型小，所以一定存在swap）

对于一次推理，目前的CC是CPU swap 同步加密 GPU等待 然后计算 然后CPU同步解密

实际上CPU加解密和GPU计算是可以并行的

需要预测GPU后续需要哪些数据 让CPU提前加密传输过去

只关注模型参数卸载和kv cache swap

如何预测后面要用哪个？（swap-in）

app使用的pattern是固定的：

  • 大模型推理 FlexGen 需要卸载模型参数 pattern是重复的 layer1 layer2 layer3
  • 中等模型推理 vLLM 需要swap kv cache 使用FIFO和LIFO
  • chunk大小一般大于128K 有明显特征

具体实现

cuda library中 cudaMemcpyAsync中

判断if (enable_encrypt && kind == cudaMemcpyHostToDevice && count >= encrypt_threshold_sz)

进行预测

否则用原来的同步copy

是一个tuple 分为src和size 判断重复只需要判断record是不是一样

  1. 修改cuda libaray 对用户透明
  2. cudaMemcpyAsync中 根据不同workload提前分析的pattern 把同步改为异步
  3. CPU异步加解密和GPU计算并行
  4. overhead可以从50%-80%降低到19%
