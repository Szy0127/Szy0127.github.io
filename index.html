CortenMM: Efficient Memory Management with Strong Correctness Guarantees

Posted on 2026-01-31 In papers Word count in article: 841 Reading time ≈ 3 mins.

CortenMM: Efficient Memory Management with Strong Correctness Guarantees

SOSP‘25 Best Paper

Background

VMA

内存管理上 Linux采用VMA+页表的两层抽象，在内存操作上需要两种类型的锁，限制了可扩展性

  • 可扩展性差
  • 并发Bug多（CVE）

  • 两个地址范围不相交的page fault 在页表层面不存在冲突，但是如果处于同一段VMA中 就会因为抢VMA锁导致竞争
  • Real-world application中存在各种mmap mprotect的调用 比较复杂

Asterinas

星绽操作系统

Design

Key insight：

  1. 两层抽象的设计是由于历史原因，目前大部分现代ISA和MMU采用相似的多级页表设计
  2. 为了实现On-demand paging COW swap等高级的内存管理语义，并不需要多一层抽象

  • 通过页表页的物理地址 可以找到page descriptor
  • 有且仅有每个页表页存在锁
  • 通过page descriptor 找到per-PTE的metadata array 存放其他信息

从这个设计上能做的简单推论：

  1. 锁的粒度变细（2M粒度） 可扩展性变好
  2. On-demand paging需要复制整个页表 会比复制VMA慢 （mmap / fork）

Lock protocol

CortenMM-rw 读写锁：在遍历页表的过程中拿读锁 找到对应的页表页以后拿写锁

读锁之间仍然有竞争（原子变量）

CoartenMM-adv：RCU+互斥锁

遍历页表的时候不加锁 只记录RCU状态

内存接口实现

Verification

用Verus(半自动的基于SMT的验证器) 进行形式化验证

  1. locking protocol正确，即 两个线程并发访问范围相交的地址 不可能同时拿到锁
  2. 对页表的修改正确（map unmap mark），并且页表结构正确（始终处于合法状态）

Evaluation

  • AMD EPYC 9965
  • 192-core CPU * 2 = 384 cores
  • 512GB DRAM * 2 = 1T

microbenchmark

单线程

  • Linux只需要创建VMA
  • Cortenmm需要分配、创建页表页

=> mmap慢

发生page fault Linux仍然需要创建页表

=> mmap+PF 快

  • Fork：-17.7%
  • fork+exec ：+23%

多线程

  • low-contention:每个线程在私有内存区域上访问
  • high-contention：每个线程在共享内存上随机访问

  • cortenmm几乎线性增长，384thread情况下 PF快33倍 unmap-virt快2270倍
  • Cortenmm 64thread开始掉的原因：最后一层页表页发生竞争

Real-world benchmark

cortenmm减少了并发情况下 application在page fault/unmap/mprotect上的耗时

在某些应用场景下，使用更先进的内存分配器（tcmalloc） 可以大幅度提升Linux的可扩展性（通过减少对unmap等syscall的调用）

但是使用tcmalloc会造成其他副作用 比如占用内存变多

Conclusion

CortenMM简化了内存管理系统，摈弃了传统的VMA，而使用单层抽象，同时提出了一种面向 MMU 编程的事务型接口，并配套设计了相应的加锁协议。这些加锁协议避免了在软件抽象层上产生不必要的竞争，在维持事务原子性的同时显著提升了性能和可扩展性。加锁协议和事务接口中的基本功能进行了形式化验证，从而提供了强有力的一致性保障。在真实世界基准测试中， CortenMM 性能比 Linux 高出 1.2 倍至 26 倍。

TZ-LLM: Protecting On-Device Large Language Models with Arm TrustZone

Posted on 2026-01-31 In papers Word count in article: 908 Reading time ≈ 3 mins.

TZ-LLM: Protecting On-Device Large Language Models with Arm TrustZone

EuroSys‘26

Background

现代移动应用能够自动将用户数据整合到大语言模型的提示中，以生成个性化的响应。与使用云端系统不同，当前出现了一种日益增长的趋势，即将 LLM 推理任务直接部署在移动设备上运行，因为这样做可以消除查询云服务所带来的网络延迟，并将用户的私有数据保留在其本地设备上。

然而，在移动设备上存储 LLM 模型参数也带来了专有模型泄露给不可信用户的风险，因为移动设备容易遭受越狱攻击。模型泄露可能给模型提供方造成重大经济损失，因为此类模型的开发成本可能高达数百万美元。此外，泄露还可能严重削弱模型提供方在竞争激烈的大语言模型市场中的竞争优势。

根据先前的研究，大多数移动应用对其设备端模型完全未加保护，而其他一些应用虽然对模型文件进行了加密，但仍允许攻击者从内存中提取出明文形式的模型参数。

Challenge

Memory

存在Memory efficiency和fast inference的trade off

大模型推理需要内存太大，如果动态扩缩容，就会面临冷启动的问题。

NPU

REE/TEE缺少安全NPU共享的方案

  • NPU在启动时默认配置为非安全设备
  • Rockchip NPU将llama3-8B模型的prefill提高12.5倍 decode提高1.3倍
  • 使用两个驱动，REE/TEE中detach/attach 进行完整的切换，一次需要32ms 太慢

Design

  • TA：llama.cpp
  • REE OS:OpenHarmony v4.1 with Linux v5.10
  • TEE OS: OpenHarmony OS and chcore

Pipelined Parameter Restoration

相对于传统的纯计算操作，额外增加了三个算子

  • Allocation (CPU)
  • Loading (IO)
  • Decryption (CPU)

优先级调度策略：

  1. Computation
  2. Decryption
  3. Allocation
  4. Loading

允许CPU抢占

Pipeline-Aware Secure Memory Management

两个TZASC region：

  • LLM参数
  • KV cache, activations,and other data

通过CMA实现连续内存获取

TEE-REE NPU Time-Sharing

Data plane负责：

  1. 初始化job上下文，包括IOPT内存 register buffer等
  2. MMIO提交job到硬件
  3. 处理中断

Control plane负责：

  1. 提交job到软件队列
  2. 调度job
  3. 通过data plane提交job到硬件
  4. 调度下一个job

  • REE kernel中放完整的NPU driver 包括调度功能
  • TEE 中放部分NPU driver 只有构造job和提交job到硬件

NPU multiplexing实现：

  1. TA有一个乘法任务 提交到TEE NPU driver
  2. TEE NPU driver构造一个secure job 通过tzdriver （smc）通知REE NPU driver
  3. REE NPU driver把这个secure job加入调度队列
  4. REE NPU driver进行正常的调度和计算
  5. REE NPU driver调度到一个secure job
  6. 通过tzdriver 控制流返回到TEE NPU driver
  7. TEE NPU driver 配置TZPC TZASC GIC 把NPU切换为secure mode
  8. TEE NPU driver通过MMIO提交secure job到硬件
  9. TEE NPU driver等待任务完成 配置TZPC TZASC GIC 把NPU 切换为normal mode

Evaluation

  1. TTFT

  • Strawmen baseline: 77.1%-91.1% improvement
  • REE-Flash baseline：2.5%∼22.3%, 22.2%∼55.3%, 10.2%∼15.2% overhead
  • REE-Memory baseline: up to 8.5× overhead (parameter restoration)

prompt短的情况下 overhead非常明显

  1. Decode

  • Straewman baseline: 0.9%∼23.2% improvement
  • REE baseline:1.3%∼4.9% overhead (NPU multiplexing)
  1. NPU

  • 额外3.8% and 3.0% 的overhead
  • smc switch + TZASC TZPC配置+GIC配置耗时在prefill decode阶段占比为1.6%∼2.7% and 2.3%∼5.7%

Drawback

Secure memory需要CMA分配连续内存

  1. 耗时久

  1. 影响其他应用性能

  1. 需要改EL3

Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

NDSS’25

针对端侧DNN模型的隔离环境

Background

DNN模型隐私保护

  • 保护知识产权
  • 保护训练数据信息

现有TEE方案（TrustZone）的缺陷：

  • 内存有限
  • 无法安全访问加速器（部分卸载到REE则会容易被攻击，加密或混淆则开销大）
  • 需要TEE-OS，与加速器软件栈不兼容

Design Goal

本工作提出ASGARD：

  • 虚拟化隔离（pkvm）
  • 加速器直通给虚拟机 （VFIO）
  • 改EL2 不改EL3
  • Guest用Linux 加速器驱动不需要改

问题和挑战：

  1. 基于虚拟化的TEE方案没有安全IO直通的方案
  2. IOMMU驱动和依赖都在TCB
  3. 虚拟化给推理带来性能影响

  1. 启动平台 验证（pkvm）
  2. 启动enclave （pkvm）
  3. DNN模型配置 （理论上需要加解密 但是本工作没有涉及）
  4. 加速器分配 （配置MMU与IOMMU页表）
  5. DNN推理服务 （通过vsock和共享内存通信）

EL2的TEEvisor保证：

  1. 对于NPU的访问必须经过授权，未经授权的Host无法直接通过NPU接口来窃取数据。
  2. NPU对enclave memory的访问必须经过授权，当Host占有NPU时无法通过IOMMU访问enclave内存

Design Details & Evaluation

  • RK3588S （Armv8.2-A SoC RockchipNPU）
  • PKVM

Secure Accelerator I/O Passthrough

安全保证：

  • 有且仅有TEEVisor对敏感接口有控制权
  • 加速器任何时刻只分给一个enclave
  • enclave使用完后，加速器无残留状态和数据

TEEvisor保证控制：

  • stage2 MMU
  • 所有IOMMU的寄存器地址范围
  • 加速器敏感接口

Guest需要用或者释放NPU的时候hypercall到pkvm EL2会执行flush iotlb等操作

加速器使用过程：

  • 初始化：配置所有IOMMU，只有加速器的IOMMU可以访问enclave内存；enclave访问加速器MMIO区域
  • 切换：重新配置MMU IOMMU 需要刷tlb iotlb。需要加速器设备支持reset 接口
  • 回收：REE可以任意时候回收，TEE 清除数据 改enclave stage2页表 IOMMU页表

切换开销：

推理latency（224x224图像分类任务）

  • TrustZone方案需要混淆，性能差
  • ASGARD对REE方案overhead只有2.01%

TCB and Attack Surface Reduction

  • IOMMU 电源 时钟管理给REEVisor。当REE关闭IOMMU 对应的设备自动关了 不会绕过
  • 不需要guest kernel中的其他进程间隔离方案，禁用Seccomp SELinux
  • 去掉没用的kernel features 定制rootfs ，大部份busybox的可执行文件是不需要的

DNN Inference Latency Optimization

CPU-fallback operations 当加速器不支持此算子 会通过触发中断 -> vmexit 很慢

优化方案：分析计算图 把不存在依赖的需要fallback的算子集中到一起 用一个vm exit

把原来的计算图拆成一个个子图 每个子图要么全是不需要fallback的 要么全是需要fallback的 这样一个子图只需要fallback一次

多个fallback算子没有依赖，但是由于默认DFS执行的顺序，会先后执行。可以切分子图，把这些没有依赖的fallback划分到一个子图里

具体实现上：

rknn的编译器和runtime都是闭源的，所以计算图是静态划分的，分成多个.rknn model文件

需要在推理框架中手动初始化各个模型文件 并且分开推理 然后整合中间数据

  • NPU 调rknn_run
  • CPU 调TensorFlow Lite runtime

input：16 tokens，执行一次生成过程，统计latency

对REE来说，一次推理需要1ms，遇到38次fallback 中断。

对ASAGARD 38*0.03=1.14ms 和推理时间一个量级

观察内存的使用，exit-coalescing的优化并不会对内存占用产生负面影响（峰值相同）

Conclusion

  • 提出NPU安全直通方案，主要是由TEEvisor进行MMIOregion和页表的切换
  • 识别出DNN推理过程中虚拟化带来的开销来源（CPU-fallback算子导致的vm-exit），并通过划分计算子图的方式进行优化。

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

Posted on 2026-01-31 In papers Word count in article: 1k Reading time ≈ 4 mins.

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

CCS24

Background

Linux内核庞大且复杂，其中很多功能对于机密计算应用来说都是冗余的，除了增大TCB与攻击面之外没有意义。

CVM攻击面包括：

  • Shared memory
  • PIO MMIO
  • Hypercall
  • Interrupts exceptions
  • CPUID leaves

Heckler(Security’24) 是已经公开的利用int 0x80恶意中断进行攻击的例子

Design

Gramine-TDX是一个轻量级的LibOS（完全代替Linux作为guest kernel）

Hypervisor使用QEMU/KVM

LibOS

  • 不支持多进程 fork 没有IPC
  • 不能用硬件 例如GPU
  • 不支持legacy 16/32bit 只支持64bit
  • 1:1 mapping
  • 没有network stack和fs stack
  • 不需要修改APP，不限语言 例如C, C++, Rust, Python, Java, Go
  • 有且仅有三个virtio驱动，并且完全按照TDX威胁模型重写，过滤不合法输入

Deployment Model

不需要虚拟机镜像，需要的文件直接从主机文件系统读取

  • 配置信息 （manifest）
  • 共享库（提前准备好的一份glibc patch过的）
  • 其他文件 提前加密好 gramine负责解密

不用虚拟网卡 不用网络配置

Chain of Trust

  • bios -> MRTD
  • gramine kernel二进制->RTMR1
  • 配置文件manifest 其中包括了其他普通文件的哈希 RTMR2

td启动时tdx module会验证二进制，记录到这些寄存器中，之后提供给远程验证

对于RTMR2 td可以自己extend定制化的内容（使用tdcall请求tdx module）

Implementation

manifest中指定entry point app hard-code写死输入 防止attacker通过verbose等泄漏信息

提供一个patch过的libc 用于适配gramine的内核，提前编译成动态库，然后mount给虚拟机

virtio-fs

本身用于host-guest共享文件 host是fuse server，guest是fuse client

gramine-tdx在host上用virtiofsd

文件请求通过fuse与virtio直接转到host，因此guest不需要fs stack

intel protected fs

intel基于自己的硬件已经实现了一套软件的加密fs协议 host上的文件数据是加密的 读取的时候gramine-tdx kernel自动透明地会进行解密

  • manifest中 fs.mount type=”encrypted”
  • manifest中sgx.trusted_files 可以记录file和对应的sha256 检察 完整性 （manifest的完整性也会被检察）

virtio-vsock

本身用于host-guest传输网络包

  • guest上和host上都用socket api 同一套地址 就可以通信
  • host/guest上都可以用socat进行socket包和TCP包的转化
  1. guest内app全都不需要改 正常通过端口提供服务 例如8080
  2. gramine-tdx的kernel实现中会进行网络包的转化，并通过virtio-vsock转交给host
  3. host上socat TCP4-LISTEN:8080,reuseaddr,fork VSOCK-CONNECT:123:1234 可以在host上的8080访问guest内的服务

sum

Virtio-fs Virtio-vsock
功能 文件 网络
一般方案 virtio-blk virtio-net
一般情况 virtio->guest fs stack->块设备 virtio->guest network stack->NIC
一般使用 qemu virtio device 然后在guest中mount qemu virtio device 然后用socket api 或者用socat
Gramine使用 guest侧mount写在kernel中 host用virtiofsd guest侧socat写在kernel中 host用socat
Gramine virtio->host fuse->host fs stack virtio->host socat->host network stack

减小TCB 减小攻击面 但是fuse和socat会带来overhead

Security Analysis

基本就是控制input、检查input合法

Evaluation

Testbed：

  • Intel Xeon Platinum 8570 CPU , 56 cores , 1 TB (16 channels × 64 GB/DIMM) DRAM
  • Host：Ubuntu 24.04, Intel TDX-enabled Linux kernel v6.8.
  • Intel TDX-patched QEMU v8.2.
  • Guest： Ubuntu 24.04,intel TDX-enabled guest Linux v6.8
  • Intel TDX Module v1.5 (build_num 698).
  • vCPU数量与Application需要线程数量相等

Varaints

Microbenchmarks

In-memory 计算开销小于5%

IO性能差，在于gramine-tdx中的virtio-fs驱动实现过于简单

AI Frameworks

  • PyTorch: AlexNet for image classification.
  • OpenVINO,TensorFlow:ResNet-50 for inference
  • TensorFlow:BERT for inference
  • Candle:LLaMA

大部分情况下几乎和native一致，平均性能overhead在10%

其他Application例如databse和web server，由于需要socat的转发，并且gramine-tdx的virito-vsock设计过于简单，性能会降至10%

Conclusion

Gramine-TDX为TDX机密虚拟机定制了guest kernel，大幅减小了攻击面，提升了安全性。

由于重点在安全和代码质量上，性能方面看来目前只适合不需要网络的纯计算任务（例如模型推理）

Effective Modern C++

型别推导

  1. 模板类型推导

    模板T的类型与ParamType(修饰)是有关的

    • 无修饰：T不会带const 引用
    • 参数是指针/引用：T不带引用 如果参数修饰带const则T不带const
    • 参数是万能引用(T&&)：T会和参数类型和传入的参数类型一模一样 带任何修饰 除了参数是右值的情况

  2. auto的类型推导与模板一模一样

    auto&& 万能引用（带修饰 左值->左值 右值->右值）

    直接用auto 不会带const 不会带引用（除非右边取地址）

  3. decltype 带修饰、引用、模板类型，一模一样

    decltype(auto)可以保证引用不被去除

    特例：decltype((x))是int&

  4. boost::typeindex::type_id_with_cvr<decltype(x)>().pretty_name()可以精确地输出类型

    IDE对于复杂类型不一定准确

auto

  1. auto声明变量的好处：

    • 通过编译器优化时间空间上的处理（lambda表达式返回值是std::function<>）

    • 避免自己搞错类型造成bug

    • 避免const带来的额外复制开销

    副作用是没法一眼看出来类型 但是可以用过好的IDE缓和

    某些情况下大括号小括号初始化有区别，例如std::vector

    可以使用auto先声明一个初始化列表 然后传入小括号

  3. nullptr代替0或NULL

  4. using替代typedef

    转型不是只改变解释方式不改变数据，转型会生成代码（int和double的表示方式都不同）

    • static_cast 强迫转型 但是在类的成员函数中转成基类调基类的函数会导致生成函数副本，使用错误的this
    • dynamic_cast 向下转型，可以通过基类派生类，但是效率不高，最好用virtual实现相同的功能

    • 操作了private成员/声明为const的函数/变量实际修改了内容

    • 不抛异常

    • 序列容器：用返回值更新迭代器
    • 关联容器：迭代器后缀自增

  10. 自定义allocater需要注意很多东西，例如接受的参数是T的个数，返回T*

  11. 自定义allocater的好处有 单线程时避免线程同步的开销、放入同一块堆等

  1. 相等是数据相等 等价是自己定义的函数 成员函数与stl非成员函数方法不同

  2. 存指针需要指定比较函数 否则根据十六进制比较

函数

  1. STL用到的函数都是按值传递的，因此需要保证小巧和单态

  2. 判别式（predicate）：返回bool的函数

    纯函数：结果仅与参数与常数有关

在程序中使用STL

  1. 用算法避免循环 例如insert find for_each

    • 循环需要保证迭代器有效
    • 算法实现者可以通过容器的实现细节提高遍历效率
    • 算法名称表明意图

  2. 选择容器的成员函数而不是普通的STL算法

    容器的成员函数是针对容器实现的 效率和正确性都有保证

  3. 查找：

    未排序：count 遍历所有 返回个数；find 找到第一个就返回迭代器 相等性

    排序：lower_bound upper_bound equal_range 等价性

    count = equal_range + distance

  4. STL组件传入函数对象而不是函数

    函数对象的()方法可以内联直接被编译器使用

    函数只能作为指针 每次都调用

  5. 使用STL算法会带来很多的嵌套调用 难以阅读 最好分步 但是仍不用循环
