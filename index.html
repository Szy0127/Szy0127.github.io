TZ-LLM: Protecting On-Device Large Language Models with Arm TrustZone

Posted on 2026-02-02 Word count in article: 892 Reading time ≈ 3 mins.

TZ-LLM: Protecting On-Device Large Language Models with Arm TrustZone

EuroSys‘26

Background

现代移动应用能够自动将用户数据整合到大语言模型的提示中，以生成个性化的响应。与使用云端系统不同，当前出现了一种日益增长的趋势，即将 LLM 推理任务直接部署在移动设备上运行，因为这样做可以消除查询云服务所带来的网络延迟，并将用户的私有数据保留在其本地设备上。

然而，在移动设备上存储 LLM 模型参数也带来了专有模型泄露给不可信用户的风险，因为移动设备容易遭受越狱攻击。模型泄露可能给模型提供方造成重大经济损失，因为此类模型的开发成本可能高达数百万美元。此外，泄露还可能严重削弱模型提供方在竞争激烈的大语言模型市场中的竞争优势。

根据先前的研究，大多数移动应用对其设备端模型完全未加保护，而其他一些应用虽然对模型文件进行了加密，但仍允许攻击者从内存中提取出明文形式的模型参数。

Challenge

Memory

存在 Memory efficiency 和 fast inference 的 trade off

大模型推理需要内存太大，如果动态扩缩容，就会面临冷启动的问题。

NPU

REE/TEE 缺少安全 NPU 共享的方案

  • NPU 在启动时默认配置为非安全设备
  • Rockchip NPU 将 llama3-8B 模型的 prefill 提高 12.5 倍 decode 提高 1.3 倍
  • 使用两个驱动，REE/TEE 中 detach/attach 进行完整的切换，一次需要 32ms 太慢

Design

  • TA：llama.cpp
  • REE OS:OpenHarmony v4.1 with Linux v5.10
  • TEE OS: OpenHarmony OS and chcore

Pipelined Parameter Restoration

相对于传统的纯计算操作，额外增加了三个算子

  • Allocation (CPU)
  • Loading (IO)
  • Decryption (CPU)

优先级调度策略：

  1. Computation
  2. Decryption
  3. Allocation
  4. Loading

允许 CPU 抢占

Pipeline-Aware Secure Memory Management

两个 TZASC region：

  • LLM 参数
  • KV cache, activations,and other data

通过 CMA 实现连续内存获取

TEE-REE NPU Time-Sharing

Data plane 负责：

  1. 初始化 job 上下文，包括 IOPT 内存 register buffer 等
  2. MMIO 提交 job 到硬件
  3. 处理中断

Control plane 负责：

  1. 提交 job 到软件队列
  2. 调度 job
  3. 通过 data plane 提交 job 到硬件
  4. 调度下一个 job

  • REE kernel 中放完整的 NPU driver 包括调度功能
  • TEE 中放部分 NPU driver 只有构造 job 和提交 job 到硬件

NPU multiplexing 实现：

  1. TA 有一个乘法任务 提交到 TEE NPU driver
  2. TEE NPU driver 构造一个 secure job 通过 tzdriver （smc）通知 REE NPU driver
  3. REE NPU driver 把这个 secure job 加入调度队列
  4. REE NPU driver 进行正常的调度和计算
  5. REE NPU driver 调度到一个 secure job
  6. 通过 tzdriver 控制流返回到 TEE NPU driver
  7. TEE NPU driver 配置 TZPC TZASC GIC 把 NPU 切换为 secure mode
  8. TEE NPU driver 通过 MMIO 提交 secure job 到硬件
  9. TEE NPU driver 等待任务完成 配置 TZPC TZASC GIC 把 NPU 切换为 normal mode

Evaluation

  1. TTFT

  • Strawmen baseline: 77.1%-91.1% improvement
  • REE-Flash baseline：2.5%∼22.3%, 22.2%∼55.3%, 10.2%∼15.2% overhead
  • REE-Memory baseline: up to 8.5× overhead (parameter restoration)

prompt 短的情况下 overhead 非常明显

  1. Decode

  • Straewman baseline: 0.9%∼23.2% improvement
  • REE baseline:1.3%∼4.9% overhead (NPU multiplexing)
  1. NPU

  • 额外 3.8% and 3.0% 的 overhead
  • smc switch + TZASC TZPC 配置 +GIC 配置耗时在 prefill decode 阶段占比为 1.6%∼2.7% and 2.3%∼5.7%

Drawback

Secure memory 需要 CMA 分配连续内存

  1. 耗时久

  1. 影响其他应用性能

  1. 需要改 EL3

CortenMM: Efficient Memory Management with Strong Correctness Guarantees

Posted on 2026-02-02 Word count in article: 822 Reading time ≈ 3 mins.

CortenMM: Efficient Memory Management with Strong Correctness Guarantees

SOSP‘25 Best Paper

Background

VMA

内存管理上 Linux 采用 VMA+ 页表的两层抽象，在内存操作上需要两种类型的锁，限制了可扩展性

  • 可扩展性差
  • 并发 Bug 多（CVE）

  • 两个地址范围不相交的 page fault 在页表层面不存在冲突，但是如果处于同一段 VMA 中 就会因为抢 VMA 锁导致竞争
  • Real-world application 中存在各种 mmap mprotect 的调用 比较复杂

Asterinas

星绽操作系统

Design

Key insight：

  1. 两层抽象的设计是由于历史原因，目前大部分现代 ISA 和 MMU 采用相似的多级页表设计
  2. 为了实现 On-demand paging COW swap 等高级的内存管理语义，并不需要多一层抽象

  • 通过页表页的物理地址 可以找到 page descriptor
  • 有且仅有每个页表页存在锁
  • 通过 page descriptor 找到 per-PTE 的 metadata array 存放其他信息

从这个设计上能做的简单推论：

  1. 锁的粒度变细（2M 粒度） 可扩展性变好
  2. On-demand paging 需要复制整个页表 会比复制 VMA 慢 （mmap / fork）

Lock protocol

CortenMM-rw 读写锁：在遍历页表的过程中拿读锁 找到对应的页表页以后拿写锁

读锁之间仍然有竞争（原子变量）

CoartenMM-adv：RCU+ 互斥锁

遍历页表的时候不加锁 只记录 RCU 状态

内存接口实现

Verification

用 Verus(半自动的基于 SMT 的验证器) 进行形式化验证

  1. locking protocol 正确，即 两个线程并发访问范围相交的地址 不可能同时拿到锁
  2. 对页表的修改正确（map unmap mark），并且页表结构正确（始终处于合法状态）

Evaluation

  • AMD EPYC 9965
  • 192-core CPU * 2 = 384 cores
  • 512GB DRAM * 2 = 1T

microbenchmark

单线程

  • Linux 只需要创建 VMA
  • Cortenmm 需要分配、创建页表页

=> mmap 慢

发生 page fault Linux 仍然需要创建页表

=> mmap+PF 快

  • Fork：-17.7%
  • fork+exec ：+23%

多线程

  • low-contention:每个线程在私有内存区域上访问
  • high-contention：每个线程在共享内存上随机访问

  • cortenmm 几乎线性增长，384thread 情况下 PF 快 33 倍 unmap-virt 快 2270 倍
  • Cortenmm 64thread 开始掉的原因：最后一层页表页发生竞争

Real-world benchmark

cortenmm 减少了并发情况下 application 在 page fault/unmap/mprotect 上的耗时

在某些应用场景下，使用更先进的内存分配器（tcmalloc） 可以大幅度提升 Linux 的可扩展性（通过减少对 unmap 等 syscall 的调用）

但是使用 tcmalloc 会造成其他副作用 比如占用内存变多

Conclusion

CortenMM 简化了内存管理系统，摈弃了传统的 VMA，而使用单层抽象，同时提出了一种面向 MMU 编程的事务型接口，并配套设计了相应的加锁协议。这些加锁协议避免了在软件抽象层上产生不必要的竞争，在维持事务原子性的同时显著提升了性能和可扩展性。加锁协议和事务接口中的基本功能进行了形式化验证，从而提供了强有力的一致性保障。在真实世界基准测试中， CortenMM 性能比 Linux 高出 1.2 倍至 26 倍。

Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

Posted on 2026-01-31 In papers Word count in article: 1k Reading time ≈ 4 mins.

Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

NDSS’25

针对端侧DNN模型的隔离环境

Background

DNN模型隐私保护

  • 保护知识产权
  • 保护训练数据信息

现有TEE方案（TrustZone）的缺陷：

  • 内存有限
  • 无法安全访问加速器（部分卸载到REE则会容易被攻击，加密或混淆则开销大）
  • 需要TEE-OS，与加速器软件栈不兼容

Design Goal

本工作提出ASGARD：

  • 虚拟化隔离（pkvm）
  • 加速器直通给虚拟机 （VFIO）
  • 改EL2 不改EL3
  • Guest用Linux 加速器驱动不需要改

问题和挑战：

  1. 基于虚拟化的TEE方案没有安全IO直通的方案
  2. IOMMU驱动和依赖都在TCB
  3. 虚拟化给推理带来性能影响

  1. 启动平台 验证（pkvm）
  2. 启动enclave （pkvm）
  3. DNN模型配置 （理论上需要加解密 但是本工作没有涉及）
  4. 加速器分配 （配置MMU与IOMMU页表）
  5. DNN推理服务 （通过vsock和共享内存通信）

EL2的TEEvisor保证：

  1. 对于NPU的访问必须经过授权，未经授权的Host无法直接通过NPU接口来窃取数据。
  2. NPU对enclave memory的访问必须经过授权，当Host占有NPU时无法通过IOMMU访问enclave内存

Design Details & Evaluation

  • RK3588S （Armv8.2-A SoC RockchipNPU）
  • PKVM

Secure Accelerator I/O Passthrough

安全保证：

  • 有且仅有TEEVisor对敏感接口有控制权
  • 加速器任何时刻只分给一个enclave
  • enclave使用完后，加速器无残留状态和数据

TEEvisor保证控制：

  • stage2 MMU
  • 所有IOMMU的寄存器地址范围
  • 加速器敏感接口

Guest需要用或者释放NPU的时候hypercall到pkvm EL2会执行flush iotlb等操作

加速器使用过程：

  • 初始化：配置所有IOMMU，只有加速器的IOMMU可以访问enclave内存；enclave访问加速器MMIO区域
  • 切换：重新配置MMU IOMMU 需要刷tlb iotlb。需要加速器设备支持reset 接口
  • 回收：REE可以任意时候回收，TEE 清除数据 改enclave stage2页表 IOMMU页表

切换开销：

推理latency（224x224图像分类任务）

  • TrustZone方案需要混淆，性能差
  • ASGARD对REE方案overhead只有2.01%

TCB and Attack Surface Reduction

  • IOMMU 电源 时钟管理给REEVisor。当REE关闭IOMMU 对应的设备自动关了 不会绕过
  • 不需要guest kernel中的其他进程间隔离方案，禁用Seccomp SELinux
  • 去掉没用的kernel features 定制rootfs ，大部份busybox的可执行文件是不需要的

DNN Inference Latency Optimization

CPU-fallback operations 当加速器不支持此算子 会通过触发中断 -> vmexit 很慢

优化方案：分析计算图 把不存在依赖的需要fallback的算子集中到一起 用一个vm exit

把原来的计算图拆成一个个子图 每个子图要么全是不需要fallback的 要么全是需要fallback的 这样一个子图只需要fallback一次

多个fallback算子没有依赖，但是由于默认DFS执行的顺序，会先后执行。可以切分子图，把这些没有依赖的fallback划分到一个子图里

具体实现上：

rknn的编译器和runtime都是闭源的，所以计算图是静态划分的，分成多个.rknn model文件

需要在推理框架中手动初始化各个模型文件 并且分开推理 然后整合中间数据

  • NPU 调rknn_run
  • CPU 调TensorFlow Lite runtime

input：16 tokens，执行一次生成过程，统计latency

对REE来说，一次推理需要1ms，遇到38次fallback 中断。

对ASAGARD 38*0.03=1.14ms 和推理时间一个量级

观察内存的使用，exit-coalescing的优化并不会对内存占用产生负面影响（峰值相同）

Conclusion

  • 提出NPU安全直通方案，主要是由TEEvisor进行MMIOregion和页表的切换
  • 识别出DNN推理过程中虚拟化带来的开销来源（CPU-fallback算子导致的vm-exit），并通过划分计算子图的方式进行优化。

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

Posted on 2026-01-31 In papers Word count in article: 1k Reading time ≈ 4 mins.

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

CCS24

Background

Linux内核庞大且复杂，其中很多功能对于机密计算应用来说都是冗余的，除了增大TCB与攻击面之外没有意义。

CVM攻击面包括：

  • Shared memory
  • PIO MMIO
  • Hypercall
  • Interrupts exceptions
  • CPUID leaves

Heckler(Security’24) 是已经公开的利用int 0x80恶意中断进行攻击的例子

Design

Gramine-TDX是一个轻量级的LibOS（完全代替Linux作为guest kernel）

Hypervisor使用QEMU/KVM

LibOS

  • 不支持多进程 fork 没有IPC
  • 不能用硬件 例如GPU
  • 不支持legacy 16/32bit 只支持64bit
  • 1:1 mapping
  • 没有network stack和fs stack
  • 不需要修改APP，不限语言 例如C, C++, Rust, Python, Java, Go
  • 有且仅有三个virtio驱动，并且完全按照TDX威胁模型重写，过滤不合法输入

Deployment Model

不需要虚拟机镜像，需要的文件直接从主机文件系统读取

  • 配置信息 （manifest）
  • 共享库（提前准备好的一份glibc patch过的）
  • 其他文件 提前加密好 gramine负责解密

不用虚拟网卡 不用网络配置

Chain of Trust

  • bios -> MRTD
  • gramine kernel二进制->RTMR1
  • 配置文件manifest 其中包括了其他普通文件的哈希 RTMR2

td启动时tdx module会验证二进制，记录到这些寄存器中，之后提供给远程验证

对于RTMR2 td可以自己extend定制化的内容（使用tdcall请求tdx module）

Implementation

manifest中指定entry point app hard-code写死输入 防止attacker通过verbose等泄漏信息

提供一个patch过的libc 用于适配gramine的内核，提前编译成动态库，然后mount给虚拟机

virtio-fs

本身用于host-guest共享文件 host是fuse server，guest是fuse client

gramine-tdx在host上用virtiofsd

文件请求通过fuse与virtio直接转到host，因此guest不需要fs stack

intel protected fs

intel基于自己的硬件已经实现了一套软件的加密fs协议 host上的文件数据是加密的 读取的时候gramine-tdx kernel自动透明地会进行解密

  • manifest中 fs.mount type=”encrypted”
  • manifest中sgx.trusted_files 可以记录file和对应的sha256 检察 完整性 （manifest的完整性也会被检察）

virtio-vsock

本身用于host-guest传输网络包

  • guest上和host上都用socket api 同一套地址 就可以通信
  • host/guest上都可以用socat进行socket包和TCP包的转化
  1. guest内app全都不需要改 正常通过端口提供服务 例如8080
  2. gramine-tdx的kernel实现中会进行网络包的转化，并通过virtio-vsock转交给host
  3. host上socat TCP4-LISTEN:8080,reuseaddr,fork VSOCK-CONNECT:123:1234 可以在host上的8080访问guest内的服务

sum

Virtio-fs Virtio-vsock
功能 文件 网络
一般方案 virtio-blk virtio-net
一般情况 virtio->guest fs stack->块设备 virtio->guest network stack->NIC
一般使用 qemu virtio device 然后在guest中mount qemu virtio device 然后用socket api 或者用socat
Gramine使用 guest侧mount写在kernel中 host用virtiofsd guest侧socat写在kernel中 host用socat
Gramine virtio->host fuse->host fs stack virtio->host socat->host network stack

减小TCB 减小攻击面 但是fuse和socat会带来overhead

Security Analysis

基本就是控制input、检查input合法

Evaluation

Testbed：

  • Intel Xeon Platinum 8570 CPU , 56 cores , 1 TB (16 channels × 64 GB/DIMM) DRAM

  1. 模板类型推导

    模板T的类型与ParamType(修饰)是有关的

    • 无修饰：T不会带const 引用
    • 参数是指针/引用：T不带引用 如果参数修饰带const则T不带const

  2. auto的类型推导与模板一模一样

    auto&& 万能引用（带修饰 左值->左值 右值->右值）

  4. future可能会发生阻塞 导致某些析构阻塞

  5. 线程同步的方法：

    • 使用atomic<bool> flag while(!flag)：缺点 占用资源

    • 使用条件变量：缺点 需要一把互斥锁 两个线程必须竞争锁

    • 使用std::promise``std::future 一次性通信

      暂停：p.get_future().wait()

      继续：p.set_value()

  6. std::atomic多线程

    volatile特种内存 多条语句不会被编译器优化成一条

  2. emplace置入函数比push插入函数更高效（通常）

Managing threads

  1. std::thread 创建对象就会直接开始新线程。有默认构造函数 不会和任意线程绑定 需要通过move接受别的

  2. 必须使用join或者detach。 使用detach可成为守护线程，生命周期和程序一样长；为了避免exception导致没有join可以使用一个类，析构里join

    std::for_each(threads.begin(),threads.end(),std::mem_fn(&std::thread::join))

  3. std::thread默认copy参数，即使函数使用引用，如果要用引用需要手动加std::ref

  4. 如果给的参数是创建线程的函数的临时变量，函数结束变量销毁可能导致线程的指针问题

  5. 如果需要用类的函数，语法为std::thread(&Object::f,&object,a,b,c) 类似std::bind

  6. std::thread可以move不能copy 类似std::ifstream std::unique_ptr

    通过move转义控制权到另一个std::thread对象

    当出现临时的std::thread对象时会隐式move

    可以放在std::vector中

  7. std::this_thread::get_id()拿到std::thread::id 支持比较操作

  8. std::this_thread::yield()不同于sleep等信号打断或睡眠固定时间，此操作让出time slice 供cpu调度

Sharing data between threads

  1. std::mutex有成员函数lock和unlock 但是不推荐使用 因为必须记得放锁

    最好配合lock_guard等离开作用域自动放锁的对象一起用

  2. boost::shared_lock<boost::shared_mutex>访问只读对象，此时互斥锁会等待

  3. std::lock同时锁多个 避免死锁

std::adopt_lock作为参数不用等直接拿已经有的

  1. 不一定只有锁才会导致死锁，join或者wait都会，在设计时需要留意

  2. 一些避免死锁的guidelines hierarchi try_lock

  3. std::unique_lock可以用std::defer_lock作为第二个参数，先不拿锁，之后手动lock

  4. 锁的范围应该最小

std::once_flag配合std::call_once在任意多的线程中只执行一次，进行初始化的操作

  1. 允许多个同时读，但不允许多个写或读写一起,c++不提供，可以使用boost

读：boost::shared_lock<boost::shared_mutex>

写：std::unique_lock<boost::shared_mutex>

Synchronizing concurrent operations

  1. std::condition_variable

  2. std::async用法和std::thread用法一样，但不是返回线程，而是返回std::future<T>

    想开线程执行一个函数 拿到返回值 可以先构造task 然后拿到future task传给thread执行，最后从future中拿出数据

    返回类型可用std::result_of<>

  4. std::promise<T>get_future拿到std::future<T>

  5. cv和future都可以wait_for/wait_until设置timeout
