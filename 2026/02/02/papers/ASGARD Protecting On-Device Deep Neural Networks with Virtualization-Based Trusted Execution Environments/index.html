ASGARD Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

Posted on 2026-02-02 In papers Views: Word count in article: 1k Reading time ≈ 4 mins.

ASGARD Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

NDSS’25

针对端侧 DNN 模型的隔离环境

Background

DNN 模型隐私保护

  • 保护知识产权
  • 保护训练数据信息

现有 TEE 方案（TrustZone）的缺陷：

  • 内存有限
  • 无法安全访问加速器（部分卸载到 REE 则会容易被攻击，加密或混淆则开销大）
  • 需要 TEE-OS，与加速器软件栈不兼容

Design Goal

本工作提出 ASGARD：

  • 虚拟化隔离（pkvm）
  • 加速器直通给虚拟机 （VFIO）
  • 改 EL2 不改 EL3
  • Guest 用 Linux 加速器驱动不需要改

问题和挑战：

  1. 基于虚拟化的 TEE 方案没有安全 IO 直通的方案
  2. IOMMU 驱动和依赖都在 TCB
  3. 虚拟化给推理带来性能影响

  1. 启动平台 验证（pkvm）
  2. 启动 enclave （pkvm）
  3. DNN 模型配置 （理论上需要加解密 但是本工作没有涉及）
  4. 加速器分配 （配置 MMU 与 IOMMU 页表）
  5. DNN 推理服务 （通过 vsock 和共享内存通信）

EL2 的 TEEvisor 保证：

  1. 对于 NPU 的访问必须经过授权，未经授权的 Host 无法直接通过 NPU 接口来窃取数据。
  2. NPU 对 enclave memory 的访问必须经过授权，当 Host 占有 NPU 时无法通过 IOMMU 访问 enclave 内存

Design Details & Evaluation

  • RK3588S （Armv8.2-A SoC RockchipNPU）
  • PKVM

Secure Accelerator I/O Passthrough

安全保证：

  • 有且仅有 TEEVisor 对敏感接口有控制权
  • 加速器任何时刻只分给一个 enclave
  • enclave 使用完后，加速器无残留状态和数据

TEEvisor 保证控制：

  • stage2 MMU
  • 所有 IOMMU 的寄存器地址范围
  • 加速器敏感接口

Guest 需要用或者释放 NPU 的时候 hypercall 到 pkvm EL2 会执行 flush iotlb 等操作

加速器使用过程：

  • 初始化：配置所有 IOMMU，只有加速器的 IOMMU 可以访问 enclave 内存；enclave 访问加速器 MMIO 区域
  • 切换：重新配置 MMU IOMMU 需要刷 tlb iotlb。需要加速器设备支持 reset 接口
  • 回收：REE 可以任意时候回收，TEE 清除数据 改 enclave stage2 页表 IOMMU 页表

切换开销：

推理 latency（224x224 图像分类任务）

  • TrustZone 方案需要混淆，性能差
  • ASGARD 对 REE 方案 overhead 只有 2.01%

TCB and Attack Surface Reduction

  • IOMMU 电源 时钟管理给 REEVisor。当 REE 关闭 IOMMU 对应的设备自动关了 不会绕过
  • 不需要 guest kernel 中的其他进程间隔离方案，禁用 Seccomp SELinux
  • 去掉没用的 kernel features 定制 rootfs ，大部份 busybox 的可执行文件是不需要的

DNN Inference Latency Optimization

CPU-fallback operations 当加速器不支持此算子 会通过触发中断 -> vmexit 很慢

优化方案：分析计算图 把不存在依赖的需要 fallback 的算子集中到一起 用一个 vm exit

把原来的计算图拆成一个个子图 每个子图要么全是不需要 fallback 的 要么全是需要 fallback 的 这样一个子图只需要 fallback 一次

多个 fallback 算子没有依赖，但是由于默认 DFS 执行的顺序，会先后执行。可以切分子图，把这些没有依赖的 fallback 划分到一个子图里

具体实现上：

rknn 的编译器和 runtime 都是闭源的，所以计算图是静态划分的，分成多个.rknn model 文件

需要在推理框架中手动初始化各个模型文件 并且分开推理 然后整合中间数据

  • NPU 调 rknn_run
  • CPU 调 TensorFlow Lite runtime

input：16 tokens，执行一次生成过程，统计 latency

对 REE 来说，一次推理需要 1ms，遇到 38 次 fallback 中断。

对 ASAGARD 38*0.03=1.14ms 和推理时间一个量级

观察内存的使用，exit-coalescing 的优化并不会对内存占用产生负面影响（峰值相同）

Conclusion

  • 提出 NPU 安全直通方案，主要是由 TEEvisor 进行 MMIOregion 和页表的切换
  • 识别出 DNN 推理过程中虚拟化带来的开销来源（CPU-fallback 算子导致的 vm-exit），并通过划分计算子图的方式进行优化。
