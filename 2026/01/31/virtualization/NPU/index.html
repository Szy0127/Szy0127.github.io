NPU

rknpu工作流程：

  1. dts kernel driver初始化
  2. 通过固定地址MMIO访问
  3. 给用户态暴露ioctl接口

sandbox使用npu：

tz-llm sandbox内有一个去掉调度的的npu driver 初始化MMIO submit的时候写MMIO 前后需要hypercall进行页表配置 保护。 由于没有stage2 访问的直接就是MMIO地址。 要用之前 先一个hypercall交一个job到队列 然后挂起，等调度到了再回来 进行配置。更合理 调度 公平

asgard sanbox内有一个完全一样的npu driver 通过vfio进行直通 前后需要hypercall进行页表配置 保护。 有stage2 需要VMM的vfio进行stage2内存配置 达到直通MMIO的效果。 sandbox内外各自调度 配置 使用 通过hypercall 在EL2建立隔离。争抢

在某一时刻，NPU只能提供给Host或者Guest

当提供给Guest时，也就是通过MMIO正式计算job时，一定保证

  1. 事先配置好页表 NPU可以访问sandbox 内存
  2. MMIO不可被Host访问 否则Host MMIO偷数据
  3. 结束后再次配置页表 否则host下次可以通过MMIO偷sanbox内存数据

计算任务前后进行NPU的独占、释放 是必要的

tz-llm用现有的调度队列和策略 sandbox提交任务 调度的时候进行NPU切换 合理 公平

asgard sandbox用前后hypercall抢占 不公平

能否在发现后面没有normal job的时候 省略context switch 一直调度sandbox的？

tz-llm强调data plane control plane分离 实际上就是MMIO直通+hypercall protect

asgard强调NPU driver安全 实际上就是hypercall vfio的直通是自带的

为啥非得MMIO直通？ 反正用之前都要进secure visor去配置 配置前带job 配置后直接在secure visor 进行MMIO有什么问题？

数据面仍然是直通的 配置好内存 NPU可以访问内存 而不是需要多一次copy到后端 控制面还是hypercall 原来也需要

但是区别在于 原来acquire release可以粒度很大 中间做很多MMIO，现在需要一次性把信息都传入EL2 EL2在里面做

sandbox隔离 NPU必须做的配置

  1. NPU的MMIO只能sandbox访问 host不行
  2. NPU的中断必须sandbox处理
  3. NPU可以访问sandbox memory

job

CONFIG_ROCKCHIP_RKNPU_DRM_GEM 用的是DRM

/dev/dri/card0 / card1可以提供ioctl

rknpu_mem_create_ioctl 分配一个rknpu_obj 地址放到obj_addr(来自kzalloc) dma_addr(来自sg_dma_address)返回给用户 用户提供的size会调用rk_dma_heap_buffer_alloc 和dma_buf_fd

rknpu_submit_ioctl->rknpu_submit 使用struct rknpu_submit 作为arg 这个arg由runtime.so准备 但是runtime.so不开源 只提供封装好的api

rknpu_submit以后 调用rknpu_job_schedule add job到todo_list 然后rknpu_job_next

rknpu_job_next从rknpu_dev->subcore_datas[core_index]->todo_list拿到job

rknpu_job_next->rknpu_job_commit 写MMIO 真正提交给硬件

每次ioctl提交job 只会放到list上 然后调度一个进硬件跑

rknpu_irq_handler->rknpu_job_done->rknpu_job_next 结束一个任务后 硬件中断 跑下一个任务

npu_worker有一个循环 通过lock和conditional variable 从task中拿task 然后submit (ioctl)

先ioctl 从驱动拿到一个地址 这个地址就是之后submit的时候要用到的job标识

mem_allocate

  1. DRM_IOCTL_RKNPU_MEM_CREATE 提供size 接受一个fd 分配一个dma addr 给NPU用
  2. DRM_IOCTL_RKNPU_MEM_MAP
  3. 对拿到的fd调用mmap 返回mmap后的指针 mmap在driver中就是对前面create的dma addr做了一次map

用户通过这个指针可以访问模型参数

rk_cma_heap_allocate 内核需要一个struct dma_buf scatter list 源头是cma_alloc(page count)得到的struct page*

matmul_kernel对象

  1. 事先准备A B C 三个buf 准备好各种矩阵类型的 每个类型一个 rknn_mem （并行怎么办？） rknn_mem初始化的时候通过mem_allocate也就是rknpu的ioctl 初始化这块内存
  2. rknpu2_matmul_pre0->ggml_rknpu2_matmul_kernel_create 一个matmul_kernel会根据矩阵分块的情况构造npu task 每个task给出input weight 和output的rknn_mem npu_task记录3个buf的ioctl之后的dma_addr 填入npu_task的cna_desc.feature_base_addr 一系列够造出 112个64bit的npu_regs 复制到regcmd 来自mem_allocate 最后填入tasks[0].regcmd_addr=regcmd_dma npu_task最终提交tasks的obj_addr 这个rknpu_task是内核驱动提供的地址
  3. rknpu2_matmul_pre1->for_all_weights 填充inputs weights 根据fp16或者int8处理 内存复制 从ggml node的src->data复制到准备好的rknn_mem->ptr中 这个ptr来自ioctl之后fd的mmap 这部分内存可以提供给npu用

推理的时候 需要的buf都是流动的 用到的内存就放在可以复用的buf上

npu task等结构体动态创建 需要把内存从CPU上copy到 cpu npu 共用的mmap之后的共享内存上

  1. llama.cpp 读参数到CPU 内存中 正常过程
  2. 通过ioctl向NPU driver申请一部分内存 用于NPU推理 涉及到内核的dma mmap等
  3. 推理过程中 copy模型参数从CPU内存到NPU内存 之后npu直接用这里的内存计算 理论上这里可以直接改地址？
  4. 通过ioctl 拿到一个npu job 填入相关信息 很复杂 需要构造112个64 bit
  5. ioctl submit这个job进行推理 job本身自己有标识 内容都在这个job里面 包括推理的内存地址

用户态调用的时候 rknpu_dev_init->npu_open->rknpu_init-> 注册中断 内存 都需要syscall到EL1

通过chcore api注册好内存以后 npu mmio的地址会对应到一个地址 之后npu submit的时候直接写那个地址 正常驱动是devm_ioremap_resource 也就是说TEE OS里的其实也放了一个正常的NPU driver 区别只是裁剪 降级了

TEE OS并没有stage2翻译 所以这里ioremap过后 最终访问的PA 就是NPU硬件的PA 可以直接用（控制平面）
