Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

CCS24

Background

Linux内核庞大且复杂，其中很多功能对于机密计算应用来说都是冗余的，除了增大TCB与攻击面之外没有意义。

CVM攻击面包括：

  • Shared memory
  • PIO MMIO
  • Hypercall
  • Interrupts exceptions
  • CPUID leaves

Heckler(Security’24) 是已经公开的利用int 0x80恶意中断进行攻击的例子

Design

Gramine-TDX是一个轻量级的LibOS（完全代替Linux作为guest kernel）

Hypervisor使用QEMU/KVM

LibOS

  • 不支持多进程 fork 没有IPC
  • 不能用硬件 例如GPU
  • 不支持legacy 16/32bit 只支持64bit
  • 1:1 mapping
  • 没有network stack和fs stack
  • 不需要修改APP，不限语言 例如C, C++, Rust, Python, Java, Go
  • 有且仅有三个virtio驱动，并且完全按照TDX威胁模型重写，过滤不合法输入

Deployment Model

不需要虚拟机镜像，需要的文件直接从主机文件系统读取

  • 配置信息 （manifest）
  • 共享库（提前准备好的一份glibc patch过的）
  • 其他文件 提前加密好 gramine负责解密

不用虚拟网卡 不用网络配置

Chain of Trust

  • bios -> MRTD
  • gramine kernel二进制->RTMR1
  • 配置文件manifest 其中包括了其他普通文件的哈希 RTMR2

td启动时tdx module会验证二进制，记录到这些寄存器中，之后提供给远程验证

对于RTMR2 td可以自己extend定制化的内容（使用tdcall请求tdx module）

Implementation

manifest中指定entry point app hard-code写死输入 防止attacker通过verbose等泄漏信息

提供一个patch过的libc 用于适配gramine的内核，提前编译成动态库，然后mount给虚拟机

virtio-fs

本身用于host-guest共享文件 host是fuse server，guest是fuse client

gramine-tdx在host上用virtiofsd

文件请求通过fuse与virtio直接转到host，因此guest不需要fs stack

intel protected fs

intel基于自己的硬件已经实现了一套软件的加密fs协议 host上的文件数据是加密的 读取的时候gramine-tdx kernel自动透明地会进行解密

  • manifest中 fs.mount type=”encrypted”
  • manifest中sgx.trusted_files 可以记录file和对应的sha256 检察 完整性 （manifest的完整性也会被检察）

virtio-vsock

本身用于host-guest传输网络包

  • guest上和host上都用socket api 同一套地址 就可以通信
  • host/guest上都可以用socat进行socket包和TCP包的转化
  1. guest内app全都不需要改 正常通过端口提供服务 例如8080
  2. gramine-tdx的kernel实现中会进行网络包的转化，并通过virtio-vsock转交给host
  3. host上socat TCP4-LISTEN:8080,reuseaddr,fork VSOCK-CONNECT:123:1234 可以在host上的8080访问guest内的服务

sum

Virtio-fs Virtio-vsock
功能 文件 网络
一般方案 virtio-blk virtio-net
一般情况 virtio->guest fs stack->块设备 virtio->guest network stack->NIC
一般使用 qemu virtio device 然后在guest中mount qemu virtio device 然后用socket api 或者用socat
Gramine使用 guest侧mount写在kernel中 host用virtiofsd guest侧socat写在kernel中 host用socat
Gramine virtio->host fuse->host fs stack virtio->host socat->host network stack

减小TCB 减小攻击面 但是fuse和socat会带来overhead

Security Analysis

基本就是控制input、检查input合法

Evaluation

Testbed：

  • Intel Xeon Platinum 8570 CPU , 56 cores , 1 TB (16 channels × 64 GB/DIMM) DRAM
  • Host：Ubuntu 24.04, Intel TDX-enabled Linux kernel v6.8.
  • Intel TDX-patched QEMU v8.2.
  • Guest： Ubuntu 24.04,intel TDX-enabled guest Linux v6.8
  • Intel TDX Module v1.5 (build_num 698).
  • vCPU数量与Application需要线程数量相等

Varaints

Microbenchmarks

In-memory 计算开销小于5%

IO性能差，在于gramine-tdx中的virtio-fs驱动实现过于简单

AI Frameworks

  • PyTorch: AlexNet for image classification.
  • OpenVINO,TensorFlow:ResNet-50 for inference
  • TensorFlow:BERT for inference
  • Candle:LLaMA

大部分情况下几乎和native一致，平均性能overhead在10%

其他Application例如databse和web server，由于需要socat的转发，并且gramine-tdx的virito-vsock设计过于简单，性能会降至10%

Conclusion

Gramine-TDX为TDX机密虚拟机定制了guest kernel，大幅减小了攻击面，提升了安全性。

由于重点在安全和代码质量上，性能方面看来目前只适合不需要网络的纯计算任务（例如模型推理）
