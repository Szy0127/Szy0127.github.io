TZ-LLM: Protecting On-Device Large Language Models with Arm TrustZone

Posted on 2026-01-31 Views: Word count in article: 908 Reading time ≈ 3 mins.

TZ-LLM: Protecting On-Device Large Language Models with Arm TrustZone

EuroSys‘26

Background

现代移动应用能够自动将用户数据整合到大语言模型的提示中，以生成个性化的响应。与使用云端系统不同，当前出现了一种日益增长的趋势，即将 LLM 推理任务直接部署在移动设备上运行，因为这样做可以消除查询云服务所带来的网络延迟，并将用户的私有数据保留在其本地设备上。

然而，在移动设备上存储 LLM 模型参数也带来了专有模型泄露给不可信用户的风险，因为移动设备容易遭受越狱攻击。模型泄露可能给模型提供方造成重大经济损失，因为此类模型的开发成本可能高达数百万美元。此外，泄露还可能严重削弱模型提供方在竞争激烈的大语言模型市场中的竞争优势。

根据先前的研究，大多数移动应用对其设备端模型完全未加保护，而其他一些应用虽然对模型文件进行了加密，但仍允许攻击者从内存中提取出明文形式的模型参数。

Challenge

Memory

存在Memory efficiency和fast inference的trade off

大模型推理需要内存太大，如果动态扩缩容，就会面临冷启动的问题。

NPU

REE/TEE缺少安全NPU共享的方案

  • NPU在启动时默认配置为非安全设备
  • Rockchip NPU将llama3-8B模型的prefill提高12.5倍 decode提高1.3倍
  • 使用两个驱动，REE/TEE中detach/attach 进行完整的切换，一次需要32ms 太慢

Design

  • TA：llama.cpp
  • REE OS:OpenHarmony v4.1 with Linux v5.10
  • TEE OS: OpenHarmony OS and chcore

Pipelined Parameter Restoration

相对于传统的纯计算操作，额外增加了三个算子

  • Allocation (CPU)
  • Loading (IO)
  • Decryption (CPU)

优先级调度策略：

  1. Computation
  2. Decryption
  3. Allocation
  4. Loading

允许CPU抢占

Pipeline-Aware Secure Memory Management

两个TZASC region：

  • LLM参数
  • KV cache, activations,and other data

通过CMA实现连续内存获取

TEE-REE NPU Time-Sharing

Data plane负责：

  1. 初始化job上下文，包括IOPT内存 register buffer等
  2. MMIO提交job到硬件
  3. 处理中断

Control plane负责：

  1. 提交job到软件队列
  2. 调度job
  3. 通过data plane提交job到硬件
  4. 调度下一个job

  • REE kernel中放完整的NPU driver 包括调度功能
  • TEE 中放部分NPU driver 只有构造job和提交job到硬件

NPU multiplexing实现：

  1. TA有一个乘法任务 提交到TEE NPU driver
  2. TEE NPU driver构造一个secure job 通过tzdriver （smc）通知REE NPU driver
  3. REE NPU driver把这个secure job加入调度队列
  4. REE NPU driver进行正常的调度和计算
  5. REE NPU driver调度到一个secure job
  6. 通过tzdriver 控制流返回到TEE NPU driver
  7. TEE NPU driver 配置TZPC TZASC GIC 把NPU切换为secure mode
  8. TEE NPU driver通过MMIO提交secure job到硬件
  9. TEE NPU driver等待任务完成 配置TZPC TZASC GIC 把NPU 切换为normal mode

Evaluation

  1. TTFT

  • Strawmen baseline: 77.1%-91.1% improvement
  • REE-Flash baseline：2.5%∼22.3%, 22.2%∼55.3%, 10.2%∼15.2% overhead
  • REE-Memory baseline: up to 8.5× overhead (parameter restoration)

prompt短的情况下 overhead非常明显

  1. Decode

  • Straewman baseline: 0.9%∼23.2% improvement
  • REE baseline:1.3%∼4.9% overhead (NPU multiplexing)
  1. NPU

  • 额外3.8% and 3.0% 的overhead
  • smc switch + TZASC TZPC配置+GIC配置耗时在prefill decode阶段占比为1.6%∼2.7% and 2.3%∼5.7%

Drawback

Secure memory需要CMA分配连续内存

  1. 耗时久

  1. 影响其他应用性能

  1. 需要改EL3
