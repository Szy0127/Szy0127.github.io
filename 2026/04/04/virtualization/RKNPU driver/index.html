RKNPU driver

Posted on 2026-04-04 In virtualization Views: Word count in article: 3.2k Reading time ≈ 12 mins.

RKNPU driver

linux内核驱动：https://github.com/airockchip/rknn-llm/tree/main/rknpu-driver

User

用户态使用方式：https://github.com/mtx512/rk3588-npu

用户态通过内核接口申请内存

作用是分配一段dma buffer

  • 这块内存可以被CPU访问，填入需要计算的数据
  • 这块内存地址可以被提交给NPU，NPU通过dma获取数据 并且写入计算后的结果

Kernel

原版的driver基于DRM设备提供ioctl、复用DRM设备的内存管理

后续分析基于打开CONFIG_ROCKCHIP_RKNPU_DRM_GEM

先提供一个修改后的最简版本

memory

在orangepi-5max运行后调用路径如下

  1. rknpu_simple_alloc_ioctl调用dma_alloc_attrs

  2. drivers/iommu/dma-iommu.c iommu_dma_alloc iommu_dma_alloc_pages

  3. page = dma_alloc_contiguous(dev, alloc_size, gfp); CMA连续地址的首页 cpu_addr = page_address(page); 实际上就是page_to_virt 然后page_to_phys拿到pa

  4. va为00000000cf870f10 pa为0x00000001158b7000 这个dma_alloc_attrs返回的就是这个va

  5. __iommu_dma_map中alloc iova与iommu_map_atomic 得到的iova是0x00000000fffff000 这个值就是dma addr 是填入rknpu submit字段的dma_addr

  6. rknpu_gem_mmap调用dma_mmap_attrs vma=[0x7e27a3d000-0x7e27a3e000] cpu_addr=0000000

这里驱动分配内存需要完成的工作如下：

  1. kzalloc分配一个内核对象，这个rknpu task的内核对象只是内核自己用，会记录一些元数据。
  2. dma_alloc_attrs分配dma buffer，对于dma buffer的要求是size大小的连续内存，这个函数的返回值为kvaddr 也就是内核使用的虚拟地址，真正的dma addr记录在buf->dma_addr。对于dma addr在开启iommu的时候是iova，不开的时候就是pa。
  3. mmap，把这块dma buffer对应的内存映射给用户态，因为用户态不能直接用内核态的虚拟地址。

这里分配的dma buffer默认应该会通过页表的某个bit控制缓存行为，是写穿到内存的，否则会出现CPU把数据写在CPU缓存内，NPU访问主存读到旧数据的情况。

submit

通过MMIO写入rknpu硬件寄存器

  • RKNPU_OFFSET_INT_MASK用于开启中断，rknpu任务结束后硬件会给CPU发送中断
  • RKNPU_OFFSET_PC_DATA_ADDR用于写入最重要的任务地址，regcmd_addr是regcmd这个dma buffer的地址，这个dma buffer记录了这次npu计算任务的元数据，包括input weight output的dma addr。

虽然原版代码没有提供，但是实测后发现可以不依赖中断，而是通过这种方式轮询获取任务完成的信息。

probe

arch/arm64/boot/dts/rockchip/rk3588s.dtsi

这里可以看到 rknpu和rknpu_mmu的中断号是一样的，并且rknpu_mmu的mmio区域是rknpu的子集

对于rockchip rknpu_mmu是只给rknpu用的，其他设备有其他设备自己的iommu可以用

probe太长了 主要关注

  1. 注册MMIO base，把物理地址0xfdab0000映射到内核虚拟地址。
  2. 注册中断，一共3个核，每个核一个中断，中断处理函数为rknpu_irq_handler
  3. 初始化rknpu_iommu
  4. rknpu_power_on与rknpu_power_off 给设备上电

这里设备的管理是使用完一次就延迟3s后断电

实测发现在没有上电的情况下进行MMIO，设备会直接panic重启。

IOMMU

iommu的初始化在drivers/iommu/rockchip-iommu.c

对于rockchip的iommu都是使用同一套代码，所以这个文件中的probe等函数会被执行多遍，但每一遍其实是针对不同的设备。

其中存在rk_iommu_irq，也就是说当发生中断时，除了rknpu_irq_handler会被调用，rk_iommu_irq也会被调用。

实测rk_iommu_irq没有什么用，去掉也能照常运行。

rknpu的iommu在dtsi中配置为打开，所以会默认会通过iommu进行一次地址翻译。

因此在分配完dma buffer以后，需要分配一个IOVA，把IOVA-PA的映射写到iommu的页表中。

http://pro934ad4.pic3.ysjianzhan.cn/upload/Rockchip_Developer_Guide_Linux_IOMMU_CN.pdf

虽然rk3588支持40bit dma索引，但是iommu只有2级页表，支持32bit的dma addr，故打开iommu的情况下，rknpu仍然只支持4G地址索引。

drivers/rknpu/rknpu_iommu.c提供了一些封装好的方法 主要做几件事

  1. 用内核接口例如sg_xx dma_xx 分配内存
  2. 自由进行iova的分配
  3. 用iommu接口进行iommu页表配置、domain切换等

interrupt / job schedule

rknpu_job.c中实现了调度的逻辑

每次用户进行submit的ioctl时，不直接进行MMIO，而是写到内核自己的job队列中

发生中断后，表示某个NPU核上的任务完成，可以调度下一个任务

用户提交rknpu job时，可通过RKNPU_JOB_NONBLOCK控制是否阻塞等待

内核会根据情况调用rknpu_job_wait 进入循环调用wait_event_timeout 等待RKNPU_JOB_DONE 否则让出cpu

Passthrough

把RKNPU直通进虚拟机

  1. host提前分配dma buffer，建立iommu映射，然后把dma buffer对应的HPA映射给GPA，并且分配IOVA=GPA
  2. host把mmio base 0xfdab0000映射给host
  3. host unregister rknpu与rknpu iommu的中断，可以重新register一个然后通过eventfd 在kvm中注入中断，但是实际上如果不需要job调度 可以通过轮询的方式，是不需要中断的。
  4. guest中可通过open(/dev/mem,O_SYNC) mmap的方式拿到GPA对应的dma buffer，然后填入IOVA=GPA
  5. MMIO提交给npu，此时npu会照常完成运算并写入结果。

Conclusion

NPU driver需要完成

  1. 通过内核接口进行dma内存分配，并且给用户态使用。
  2. 配置中断，实现任务提交与调度逻辑。
  3. 在有IOMMU等情况下，与IOMMU配合，分配IOVA建立页表。
  4. 对设备的电源、状态检查与管理。
