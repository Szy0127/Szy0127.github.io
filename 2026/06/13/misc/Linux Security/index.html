Linux Security

bpf

场景 1：用户态的 tcpdump，需要过滤网络包，但是每次内核只做转发，需要每个网络包 copy 到用户态 效率太低，最好内核根据某个条件自由过滤网络包。

场景 2：seccomp 限制系统调用，根据定死的规则限制太不灵活，最好有一个复杂的逻辑根据参数判断是否需要过滤，由用户态提供。

本质：

  1. 操作系统内核向下提供对硬件的调用，向上对用户态提供 api，同时涉及到内存管理、调度等重要功能，但是在具体的场景和需求中，仍然是需要由用户态来决定业务逻辑的。
  2. 内核权限更高、接触数据更底层，在这里运行用户态的逻辑更有价值。
  3. 内核本身不应该随意修改，且加入大量复杂的代码。

classic-bpf/ extent bpf：

  1. 内核提供一些入口，可以运行 bpf 程序
  2. 用户态写 bpf 程序，挂载到内核运行
  3. 内核检查，然后在挂载点运行

具体实现：

  1. bpf 程序可以用一些限定的数据结构，一些寄存器，内存。相当于是另一套小的指令集（RISC ISA）这套指令集会通过解释器或者编译器翻译成具体的汇编执行。
  2. 解释执行->编译 JIT，修改挂载点的第一个指令为 int3 ->内核编译期使用 fentry
  • bpf 的挂载点提供 context 可以访问内存数据
  • 提供一些 helper 函数

LSM-BPF

https://github.com/eunomia-bpf/bpf-developer-tutorial/blob/main/src/19-lsm-connect/README.zh.md

用 ecc 编译 依赖 clang llvm

ecli 加载

需要CONFIG_LSM_BPF 和DEBUG_INFO_BPF

可以限制当前进行的某些行为 例如无法访问某个网络地址

LSM 的确是内核开发者人工选择并写死的一组安全语义检查点。BPF LSM没有消除这个限制，而是把这些固定检查点变成了可在运行时加载、验证、JIT 和卸载的可编程策略入口。

​

它的创新不是“自动发现所有需要检查的位置”，而是：

​

固定且可信的安全语义位置

+ 强类型 BTF 接口

+ verifier 隔离

+ JIT 执行

+ 动态 trampoline

+ BPF map 和 helper

+ link 生命周期

因此 BPF LSM 很灵活，但灵活性只存在于预定义 LSM hook 覆盖的安全边界内。

有区别的是 DAC 只要有 root 就最强，但是 landlock 是限制进程的 有 root 也没用
