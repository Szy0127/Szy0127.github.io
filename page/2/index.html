CortenMM: Efficient Memory Management with Strong Correctness Guarantees

Posted on 2026-02-02 In papers Word count in article: 1.5k Reading time ≈ 5 mins.

CortenMM: Efficient Memory Management with Strong Correctness Guarantees

SOSP‘25 Best Paper

Background

VMA

内存管理上 Linux 采用 VMA+ 页表的两层抽象，在内存操作上需要两种类型的锁，限制了可扩展性

  • 可扩展性差
  • 并发 Bug 多（CVE）

  • 两个地址范围不相交的 page fault 在页表层面不存在冲突，但是如果处于同一段 VMA 中 就会因为抢 VMA 锁导致竞争
  • Real-world application 中存在各种 mmap mprotect 的调用 比较复杂

Asterinas

星绽操作系统

Design

Key insight：

  1. 两层抽象的设计是由于历史原因，目前大部分现代 ISA 和 MMU 采用相似的多级页表设计
  2. 为了实现 On-demand paging COW swap 等高级的内存管理语义，并不需要多一层抽象

  • 通过页表页的物理地址 可以找到 page descriptor
  • 有且仅有每个页表页存在锁
  • 通过 page descriptor 找到 per-PTE 的 metadata array 存放其他信息

从这个设计上能做的简单推论：

  1. 锁的粒度变细（2M 粒度） 可扩展性变好
  2. On-demand paging 需要复制整个页表 会比复制 VMA 慢 （mmap / fork）

Lock protocol

CortenMM-rw 读写锁：在遍历页表的过程中拿读锁 找到对应的页表页以后拿写锁

读锁之间仍然有竞争（原子变量）

CoartenMM-adv：RCU+ 互斥锁

遍历页表的时候不加锁 只记录 RCU 状态

内存接口实现

Verification

用 Verus(半自动的基于 SMT 的验证器) 进行形式化验证

  1. locking protocol 正确，即 两个线程并发访问范围相交的地址 不可能同时拿到锁
  2. 对页表的修改正确（map unmap mark），并且页表结构正确（始终处于合法状态）

Evaluation

  • AMD EPYC 9965
  • 192-core CPU * 2 = 384 cores
  • 512GB DRAM * 2 = 1T

microbenchmark

单线程

  • Linux 只需要创建 VMA
  • Cortenmm 需要分配、创建页表页

=> mmap 慢

发生 page fault Linux 仍然需要创建页表

=> mmap+PF 快

  • Fork：-17.7%
  • fork+exec ：+23%

多线程

  • low-contention:每个线程在私有内存区域上访问
  • high-contention：每个线程在共享内存上随机访问

  • cortenmm 几乎线性增长，384thread 情况下 PF 快 33 倍 unmap-virt 快 2270 倍
  • Cortenmm 64thread 开始掉的原因：最后一层页表页发生竞争

Real-world benchmark

cortenmm 减少了并发情况下 application 在 page fault/unmap/mprotect 上的耗时

在某些应用场景下，使用更先进的内存分配器（tcmalloc） 可以大幅度提升 Linux 的可扩展性（通过减少对 unmap 等 syscall 的调用）

但是使用 tcmalloc 会造成其他副作用 比如占用内存变多

Conclusion

CortenMM 简化了内存管理系统，摈弃了传统的 VMA，而使用单层抽象，同时提出了一种面向 MMU 编程的事务型接口，并配套设计了相应的加锁协议。这些加锁协议避免了在软件抽象层上产生不必要的竞争，在维持事务原子性的同时显著提升了性能和可扩展性。加锁协议和事务接口中的基本功能进行了形式化验证，从而提供了强有力的一致性保障。在真实世界基准测试中， CortenMM 性能比 Linux 高出 1.2 倍至 26 倍。

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

Gramine-TDX: A Lightweight OS Kernel for Confidential VMs

CCS24

Background

Linux 内核庞大且复杂，其中很多功能对于机密计算应用来说都是冗余的，除了增大 TCB 与攻击面之外没有意义。

CVM 攻击面包括：

  • Shared memory
  • PIO MMIO
  • Hypercall
  • Interrupts exceptions
  • CPUID leaves

Heckler(Security’24) 是已经公开的利用 int 0x80 恶意中断进行攻击的例子

Design

Gramine-TDX 是一个轻量级的 LibOS（完全代替 Linux 作为 guest kernel）

Hypervisor 使用 QEMU/KVM

LibOS

  • 不支持多进程 fork 没有 IPC
  • 不能用硬件 例如 GPU
  • 不支持 legacy 16/32bit 只支持 64bit
  • 1:1 mapping
  • 没有 network stack 和 fs stack
  • 不需要修改 APP，不限语言 例如 C, C++, Rust, Python, Java, Go
  • 有且仅有三个 virtio 驱动，并且完全按照 TDX 威胁模型重写，过滤不合法输入

Deployment Model

不需要虚拟机镜像，需要的文件直接从主机文件系统读取

  • 配置信息 （manifest）
  • 共享库（提前准备好的一份 glibc patch 过的）
  • 其他文件 提前加密好 gramine 负责解密

不用虚拟网卡 不用网络配置

Chain of Trust

  • bios -> MRTD
  • gramine kernel 二进制->RTMR1
  • 配置文件 manifest 其中包括了其他普通文件的哈希 RTMR2

td 启动时 tdx module 会验证二进制，记录到这些寄存器中，之后提供给远程验证

对于 RTMR2 td 可以自己 extend 定制化的内容（使用 tdcall 请求 tdx module）

Implementation

manifest 中指定 entry point app hard-code 写死输入 防止 attacker 通过 verbose 等泄漏信息

提供一个 patch 过的 libc 用于适配 gramine 的内核，提前编译成动态库，然后 mount 给虚拟机

virtio-fs

本身用于 host-guest 共享文件 host 是 fuse server，guest 是 fuse client

gramine-tdx 在 host 上用 virtiofsd

文件请求通过 fuse 与 virtio 直接转到 host，因此 guest 不需要 fs stack

intel protected fs

intel 基于自己的硬件已经实现了一套软件的加密 fs 协议 host 上的文件数据是加密的 读取的时候 gramine-tdx kernel 自动透明地会进行解密

  • manifest 中 fs.mount type=”encrypted”
  • manifest 中 sgx.trusted_files 可以记录 file 和对应的 sha256 检察 完整性 （manifest 的完整性也会被检察）

virtio-vsock

本身用于 host-guest 传输网络包

  • guest 上和 host 上都用 socket api 同一套地址 就可以通信
  • host/guest 上都可以用 socat 进行 socket 包和 TCP 包的转化
  1. guest 内 app 全都不需要改 正常通过端口提供服务 例如 8080
  2. gramine-tdx 的 kernel 实现中会进行网络包的转化，并通过 virtio-vsock 转交给 host
  3. host 上 socat TCP4-LISTEN:8080,reuseaddr,fork VSOCK-CONNECT:123:1234 可以在 host 上的 8080 访问 guest 内的服务

sum

无法复制加载中的内容

减小 TCB 减小攻击面 但是 fuse 和 socat 会带来 overhead

Security Analysis

基本就是控制 input、检查 input 合法

Evaluation

Testbed：

  • Intel Xeon Platinum 8570 CPU , 56 cores , 1 TB (16 channels × 64 GB/DIMM) DRAM
  • Host：Ubuntu 24.04, Intel TDX-enabled Linux kernel v6.8.
  • Intel TDX-patched QEMU v8.2.
  • Guest： Ubuntu 24.04,intel TDX-enabled guest Linux v6.8
  • Intel TDX Module v1.5 (build_num 698).
  • vCPU 数量与 Application 需要线程数量相等

Varaints

Microbenchmarks

In-memory 计算开销小于 5%

IO 性能差，在于 gramine-tdx 中的 virtio-fs 驱动实现过于简单

AI Frameworks

  • PyTorch: AlexNet for image classification.
  • OpenVINO,TensorFlow:ResNet-50 for inference
  • TensorFlow:BERT for inference
  • Candle:LLaMA

大部分情况下几乎和 native 一致，平均性能 overhead 在 10%

其他 Application 例如 databse 和 web server，由于需要 socat 的转发，并且 gramine-tdx 的 virito-vsock 设计过于简单，性能会降至 10%

Conclusion

Gramine-TDX 为 TDX 机密虚拟机定制了 guest kernel，大幅减小了攻击面，提升了安全性。

由于重点在安全和代码质量上，性能方面看来目前只适合不需要网络的纯计算任务（例如模型推理）

ASGARD Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

Posted on 2026-02-02 In papers Word count in article: 2k Reading time ≈ 7 mins.

ASGARD Protecting On-Device Deep Neural Networks with Virtualization-Based Trusted Execution Environments

NDSS’25

针对端侧 DNN 模型的隔离环境

Background

DNN 模型隐私保护

  • 保护知识产权
  • 保护训练数据信息

现有 TEE 方案（TrustZone）的缺陷：

  • 内存有限
  • 无法安全访问加速器（部分卸载到 REE 则会容易被攻击，加密或混淆则开销大）
  • 需要 TEE-OS，与加速器软件栈不兼容

Design Goal

本工作提出 ASGARD：

  • 虚拟化隔离（pkvm）
  • 加速器直通给虚拟机 （VFIO）
  • 改 EL2 不改 EL3
  • Guest 用 Linux 加速器驱动不需要改

问题和挑战：

  1. 基于虚拟化的 TEE 方案没有安全 IO 直通的方案
  2. IOMMU 驱动和依赖都在 TCB
  3. 虚拟化给推理带来性能影响

  1. 启动平台 验证（pkvm）
  2. 启动 enclave （pkvm）
  3. DNN 模型配置 （理论上需要加解密 但是本工作没有涉及）
  4. 加速器分配 （配置 MMU 与 IOMMU 页表）
  5. DNN 推理服务 （通过 vsock 和共享内存通信）

EL2 的 TEEvisor 保证：

  1. 对于 NPU 的访问必须经过授权，未经授权的 Host 无法直接通过 NPU 接口来窃取数据。
  2. NPU 对 enclave memory 的访问必须经过授权，当 Host 占有 NPU 时无法通过 IOMMU 访问 enclave 内存

Design Details & Evaluation

  • RK3588S （Armv8.2-A SoC RockchipNPU）
  • PKVM

Secure Accelerator I/O Passthrough

安全保证：

  • 有且仅有 TEEVisor 对敏感接口有控制权
  • 加速器任何时刻只分给一个 enclave
  • enclave 使用完后，加速器无残留状态和数据

TEEvisor 保证控制：

  • stage2 MMU
  • 所有 IOMMU 的寄存器地址范围
  • 加速器敏感接口

Guest 需要用或者释放 NPU 的时候 hypercall 到 pkvm EL2 会执行 flush iotlb 等操作

加速器使用过程：

  • 初始化：配置所有 IOMMU，只有加速器的 IOMMU 可以访问 enclave 内存；enclave 访问加速器 MMIO 区域
  • 切换：重新配置 MMU IOMMU 需要刷 tlb iotlb。需要加速器设备支持 reset 接口
  • 回收：REE 可以任意时候回收，TEE 清除数据 改 enclave stage2 页表 IOMMU 页表

切换开销：

推理 latency（224x224 图像分类任务）

  • TrustZone 方案需要混淆，性能差
  • ASGARD 对 REE 方案 overhead 只有 2.01%

TCB and Attack Surface Reduction

  • IOMMU 电源 时钟管理给 REEVisor。当 REE 关闭 IOMMU 对应的设备自动关了 不会绕过
  • 不需要 guest kernel 中的其他进程间隔离方案，禁用 Seccomp SELinux
  • 去掉没用的 kernel features 定制 rootfs ，大部份 busybox 的可执行文件是不需要的

DNN Inference Latency Optimization

CPU-fallback operations 当加速器不支持此算子 会通过触发中断 -> vmexit 很慢

优化方案：分析计算图 把不存在依赖的需要 fallback 的算子集中到一起 用一个 vm exit

把原来的计算图拆成一个个子图 每个子图要么全是不需要 fallback 的 要么全是需要 fallback 的 这样一个子图只需要 fallback 一次

多个 fallback 算子没有依赖，但是由于默认 DFS 执行的顺序，会先后执行。可以切分子图，把这些没有依赖的 fallback 划分到一个子图里

具体实现上：

rknn 的编译器和 runtime 都是闭源的，所以计算图是静态划分的，分成多个.rknn model 文件

需要在推理框架中手动初始化各个模型文件 并且分开推理 然后整合中间数据

  • NPU 调 rknn_run
  • CPU 调 TensorFlow Lite runtime

input：16 tokens，执行一次生成过程，统计 latency

对 REE 来说，一次推理需要 1ms，遇到 38 次 fallback 中断。

对 ASAGARD 38*0.03=1.14ms 和推理时间一个量级

观察内存的使用，exit-coalescing 的优化并不会对内存占用产生负面影响（峰值相同）

Conclusion

  • 提出 NPU 安全直通方案，主要是由 TEEvisor 进行 MMIOregion 和页表的切换
  • 识别出 DNN 推理过程中虚拟化带来的开销来源（CPU-fallback 算子导致的 vm-exit），并通过划分计算子图的方式进行优化。

sandbox使用npu：

tz-llm sandbox内有一个去掉调度的的npu driver 初始化MMIO submit的时候写MMIO 前后需要hypercall进行页表配置 保护。 由于没有stage2 访问的直接就是MMIO地址。 要用之前 先一个hypercall交一个job到队列 然后挂起，等调度到了再回来 进行配置。更合理 调度 公平

asgard sanbox内有一个完全一样的npu driver 通过vfio进行直通 前后需要hypercall进行页表配置 保护。 有stage2 需要VMM的vfio进行stage2内存配置 达到直通MMIO的效果。 sandbox内外各自调度 配置 使用 通过hypercall 在EL2建立隔离。争抢

在某一时刻，NPU只能提供给Host或者Guest

当提供给Guest时，也就是通过MMIO正式计算job时，一定保证

  1. 事先配置好页表 NPU可以访问sandbox 内存
  2. MMIO不可被Host访问 否则Host MMIO偷数据
  3. 结束后再次配置页表 否则host下次可以通过MMIO偷sanbox内存数据

计算任务前后进行NPU的独占、释放 是必要的

tz-llm用现有的调度队列和策略 sandbox提交任务 调度的时候进行NPU切换 合理 公平

asgard sandbox用前后hypercall抢占 不公平

能否在发现后面没有normal job的时候 省略context switch 一直调度sandbox的？

tz-llm强调data plane control plane分离 实际上就是MMIO直通+hypercall protect

asgard强调NPU driver安全 实际上就是hypercall vfio的直通是自带的

为啥非得MMIO直通？ 反正用之前都要进secure visor去配置 配置前带job 配置后直接在secure visor 进行MMIO有什么问题？

数据面仍然是直通的 配置好内存 NPU可以访问内存 而不是需要多一次copy到后端 控制面还是hypercall 原来也需要

但是区别在于 原来acquire release可以粒度很大 中间做很多MMIO，现在需要一次性把信息都传入EL2 EL2在里面做

sandbox隔离 NPU必须做的配置

Effective C++

让自己习惯C++

    1. c++有4个主要的次语言
    • C 没有模板 异常 重载
    • C++ 类 封装 继承 多态 虚函数
    • Template 泛型
    • STL

    不同次语言有不同的规约，需要使用不同的处理方式（例如传值和传引用的效率）

  2. #define 直接被替换 报错时找不到信息 并且占用内存可能反而更大

    • const 取地址合法
    • enum 取地址不合法
    • inline

  3. 用const修饰变量，函数等

    返回的对象可以使用const修饰 避免 if(a*b=c)

    const可非const可重载

    非const可转型为const

  4. 类的成员变量的初始化发生在进入构造函数之前，且一定按声明的顺序

    使用初始化列表，内置类型也使用xx()的形式

构造/析构/赋值运算

  1. 默认构造函数会初始化基类和非静态成员变量，默认析构会自动调用析构（所以不用写），默认拷贝函数会复制每个非const对象

    三种情况 不会生成默认拷贝函数

    • 有引用成员变量
    • 有const成员变量
    • 基类的拷贝函数为private

  2. 不想使用默认的函数，可以声明为private

  3. 多态的基类的析构函数加virtual 避免用基类指针释放派生类对象出现内存泄漏

    虚函数使用虚函数指针管理，会增加内存占用

    如果基类的析构没有virtual 不适合派生，例如STL

  4. 在析构中catch异常，不向下传递

  5. 在派生类的构造/析构函数调用基类的构造/析构函数时，派生类的信息已经消失，所以在基类的构造/析构函数中调用虚函数是无效的

  6. 为了实现连续赋值，让operator=返回左值引用

  7. operator=的两个对象如果相同可能会出现删掉自己的问题

    解决方法如下：

    • 开头判断是否相等 如果相等直接返回 增加了取址 条件跳转 影响效率
    • 调整顺序 先改再删
    • 用swap

  8. 自定义拷贝构造函数和拷贝函数需要保证复制了每个成员变量，编译器不会提醒

    • 复制自己所有的成员变量
    • 调用基类的拷贝构造函数 XX::operator=(rhs)

    拷贝构造函数和拷贝函数重复的代码可用一个private的init函数来写

资源管理

除了内存，文件描述符，锁、连接、socket都是资源

  1. 在一个函数中，使用指针 最后delete 可能导致由于中间部分的问题而没有delete

    使用智能指针管理，借助离开函数自动调用析构函数的机制 std::auto_ptr<A> p(new A)

    智能指针默认执行delete 而不是delete[] 无法管理数组

    需要open close成对出现的资源 可以写在一个类中，close写在类的析构中，出现异常自动调用

  2. 资源管理类小心复制，可以直接禁止复制

  3. 虽然智能指针可以通过*或->像指针一样使用，但是*A与std::auto_ptr<A>类型并不相同，不能被函数接受

    智能指针提供get方法拿到原始指针，但是这样用户会觉得麻烦

    另一种方法是在类中定义隐式转换的函数 但这样会出现非预期的转换 造成bug

  4. new delete ； new[] delete[] 否则都会有bug

    • 多个构造函数用相同形式的new
    • 使用typedef 注意是否是数组

  5. new得到的指针放入指针智能时使用独立语句，否则可能导致中间步骤的失败丢失new出来的指针

设计与声明

  1. 良好的接口不应该被误用，例如返回指针会导致没有delete/delete两次/错误方式delete等

    可以返回一个智能指针，同时设置好deleter 强制让用户使用智能指针来接受

    防止同类型的参数顺序出错，可以新声明几个类型，强制让用户使用新类型

  2. 设计class应该像设计type一样 考虑创建销毁 初始化赋值 传递 合法值 转换 操作符 接口等一系列问题

  3. 函数参数传递是按值传递，会调用构造函数，效率低，尽量使用引用传递，为了防止引用传递的后果 加const

    这样做可以解决对象切割的问题，即使用基类对象接受派生类，依然可以正确找到派生类声明为virtual的函数

    内置类型比较小 用引用涉及到指针反而慢， stl容器也尽量用值传递

  4. 函数的返回值也会涉及到构造与析构的浪费，但是此处并没有办法节省（除非返回值并不是函数生成的）

    尝试方法与问题如下：

    • 栈上分配 返回引用 返回时已经销毁 拿到已经销毁的变量
    • 堆上分配 返回指针 没人delete/连续运算时丢失临时变量的指针
    • 使用static static对象只看现值 严重bug

  5. 所有类的成员变量都应该是private 需要的话设置getter和setter通过函数访问

    好处如下：

    • 提供的接口都是函数，统一使用括号访问
    • 封装 可以随意修改实现方法
    • 取消此变量后不会让客户代码与派生类代码发生破坏

  6. 封装的内容越多，可访问内容的函数越少，改动影响的部分越少，可改动的弹性越大

    所以在两个函数具有相同功能的时候，尽量使用non-member/non-friend去替代member(成员)函数

    可以使用namespace把工具函数和类放在同一区域中

    不同的源文件不能切分类，但是可以使用同一个namespace 切分工具函数

  7. 如果一个函数的所有参数都需要类型转换，应该成为non-member

    member函数一方是this 无法隐式转换

  8. ？ swap

实现

  1. 尽量延后变量定义式出现的时间，否则可能会由于中途的异常而没有使用却承担了构造析构的成本

    直接给初值，而不是初始化之后再赋值

    循环中需要的变量放在内还是外需要对比构造析构与赋值操作的时间，放内部不影响作用域及可理解性、易维护性

  2. 少转型 用新式转型替代旧式转型

    转型不是只改变解释方式不改变数据，转型会生成代码（int和double的表示方式都不同）

    • static_cast 强迫转型 但是在类的成员函数中转成基类调基类的函数会导致生成函数副本，使用错误的this
    • dynamic_cast 向下转型，可以通过基类派生类，但是效率不高，最好用virtual实现相同的功能

  3. 避免返回handles（引用，指针，迭代器）

    风险如下：

    • 操作了private成员/声明为const的函数/变量实际修改了内容
    • handles所指内容的主体已经被析构，造成空悬

    例外：string vector等容器的operator=返回的是引用

  4. 函数需要异常安全

    三个等级：

    • 基本：不破坏数据结构（例如资源泄露，打乱排序数组）
    • 强烈：保证状态不变 可以通过copy-swap实现，及创建副本 修改在副本上 然后交换 但可能效率较低 复杂程度较高；并且，如果这个函数中调用了其他函数，但是其他函数并没有强烈的异常安全保证，自己也不行。
    • 不抛异常

  5. inline 用函数本体替换每次调用的地方，编译时候完成，编译器会选择（太复杂或者virtual会拒绝）

    • 好处：减小函数调用的开销

    • 坏处：可能增大字节码，修改后必须重新编译

  6. ？传统的方式（.h/.cpp）并没有编译依存最小化，仅仅修改一部分就需要全部编译

    可通过以下两种方式解耦接口与实现：

    • handle classes：通过组合方式，类A使用类AImpl完全其所有的函数
    • interface classes：通过继承方式，类A提供接口，全是纯虚函数，类B继承类A实现这些函数。

继承与面向对象设计

  1. public继承必须满足is-a的关系（里氏替换原则）

  2. 派生类函数会遮蔽基类的函数，具体情况为基类有多个重载的函数，派生类有一个同名的函数，那么基类的多个函数全被遮蔽（与virtual无关），如果是public继承则违反了is-a的关系

    解决方法是使用using Base::f 但这样会把所有的重载的f都拿到作用域中

    如果只想用某一个，可以先用private继承，然后在同名函数中直接调用Base::f()

  3. 区分接口继承和实现继承

    用virtual 同时继承接口和实现，可以重写各自的实现，但问题是如果忘记重写了编译器并不会报错，而是会使用基类的函数

    用pure virtual 可以强制让派生类提供实现，但是可能会出现相同代码

    在基类中提供一个default版本的代码供派生类调用，但这样会多一个函数，且造成命名混乱

    解决方法：给纯虚函数提供实现，派生类通过Base::f()调用此函数

    private virtual/public non-virtual调用virtual

  6. 函数参数的缺省值是看静态类型的，所以virtual函数的参数缺省值只有基类起作用
